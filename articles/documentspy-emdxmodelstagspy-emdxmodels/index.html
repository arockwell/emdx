    documents.py: emdx Models and Database Layer¶

    Overview¶

    The emdx knowledge base uses a layered architecture where the database layer (emdx/database/) handles raw SQL operations and the models layer (emdx/models/) provides business logic and re-exports. The documents.py module is the centerpiece, managing document CRUD operations, relationships, and full-text search integration.

    Core responsibilities: - Document lifecycle: create, read, update, soft delete, restore - Parent-child relationships with lineage tracking ("supersedes", "exploration", "variant") - FTS5 full-text search integration via virtual tables - Access tracking (timestamps and counts) - Project-based scoping

    Key Files¶

    emdx/
├── models/
│   ├── types.py              # Model layer TypedDicts (tasks, categories, tags)
│   ├── documents.py          # Re-exports database layer document functions
│   ├── tasks.py              # Task operations with epic/dependency management
│   ├── tags.py               # Tag normalization and search
│   └── categories.py         # Task category management
├── database/
│   ├── types.py              # Database layer TypedDicts (documents, cascade, groups)
│   ├── documents.py          # Core document CRUD with SQL operations
│   ├── search.py             # FTS5 search queries
│   ├── cascade.py            # Cascade pipeline stage tracking
│   └── groups.py             # Document grouping and hierarchy

    Type System Architecture¶

    TypedDict Strategy¶

    As of PRs #603 and #606, the codebase has migrated from dict[str, Any] to concrete TypedDicts, providing: - Type safety: Catch key access errors at lint time - IDE autocompletion for dictionary keys - Self-documenting schemas - Refactoring confidence across the codebase

    Design principle: Types are defined close to where they're returned. Models layer types live in models/types.py, database layer types in database/types.py.

    Type System Conventions¶

    Boolean fields — SQLite stores booleans as integers (0/1):

    is_deleted: int    # NOT bool
is_active: int     # NOT bool

    Datetime fields — SQLite returns ISO8601 strings:

    created_at: str | None    # NOT datetime
updated_at: str | None    # NOT datetime

    Literal types for enums:

    TaskStatus = Literal['open', 'active', 'blocked', 'done', 'failed']
CascadeStage = Literal['idea', 'prompt', 'analyzed', 'planned', 'done']

    TypedDict inheritance:

    class EpicTaskDict(TaskDict):
    child_count: int
    children_open: int
    children_done: int

    SQL parameter lists — After audit #6440:

    params: list[str | int | None] = []  # NOT list[Any]

    SQLite Row conversion — Requires type ignores:

    return cast(TaskDict, dict(row))  # type: ignore[arg-type, return-value]

)

    FTS5 Virtual Table (Migration 002)¶

    CREATE VIRTUAL TABLE documents_fts USING fts5(
    title, content, project,
    content=documents, content_rowid=id
)

    Critical FTS5 gotcha: documents_fts is a separate virtual table, NOT a column on documents. You must JOIN it:

    -- ❌ WRONG: "no such column" error
SELECT id FROM documents WHERE documents_fts MATCH ?

-- ✅ CORRECT: JOIN pattern
SELECT d.id FROM documents d
JOIN documents_fts fts ON d.id = fts.rowid
WHERE fts.documents_fts MATCH ?

    See emdx/database/search.py for the canonical pattern (documented in CLAUDE.md known gotchas).

    save_document():

    def save_document(
    title: str,

    Originally committed after INSERT, then called add_tags_to_document() in a separate transaction — creating an atomicity bug. Fixed in PR #594 to pass the same connection:

    # Before: commit happened here, before tagging
doc_id = cursor.lastrowid
conn.commit()  # ❌ Too early
if tags:
    add_tags_to_document(doc_id, tags)  # Separate transaction

# After: commit happens after tags
doc_id = cursor.lastrowid
if tags:
    add_tags_to_document(doc_id, tags, conn=conn)  # Same transaction
conn.commit()  # ✅ Atomic

    list_documents():

    Null-Safety Improvements (PR #594)¶

    Multiple locations accessed cursor.fetchone()[0] without null checks, crashing on empty result sets:

    # Before:
return int(cursor.fetchone()[0])  # ❌ IndexError if no result

# After:
result = cursor.fetchone()
return int(result[0]) if result else 0  # ✅ Safe default

    Applied to get_or_create_tag(), merge_tags(), count_documents(), get_stats(), and 6+ other functions.

    Architecture & Design Decisions¶

    Transaction Composition Pattern¶

    Functions accept optional conn: sqlite3.Connection | None = None parameter: - When conn is provided → run in caller's transaction - When None → create own connection and auto-commit

    This enables atomic multi-step operations while maintaining backward compatibility with 70+ existing callers.

    Input Validation (PR #594)¶

    Pagination parameters:

    if limit < 0 or offset < 0:
    raise ValueError("limit and offset must be non-negative")

    Date parameters:

    if days <= 0:
    raise ValueError("days must be positive")

    Prevents malformed SQL like datetime('now', '--5 days').

    Column name whitelisting: Dynamic SQL with user-supplied column names is a SQL injection vector. Fixed via frozen sets in tasks.py, executions.py, and groups.py.

    Type Safety Migration¶

    A comprehensive audit (#6432, #6433) eliminated unnecessary Any usage:

    Before:

    def get_stats() -> dict[str, Any]

    After:

    class ExecutionStatsDict(TypedDict):
    total: int
    recent_24h: int
    running: int
    completed: int
    failed: int

def get_stats() -> ExecutionStatsDict

    Legitimate Any patterns retained: - Pre-cast intermediate dicts before cast(TypedDict, row) calls - Dynamic update kwargs with whitelist validation

    Nullable Type Discipline¶

    Analysis (#6489) validated all 124 | None annotations as necessary:

    1. Optional FK relationships — gameplan_id: int | None, parent_task_id: int | None
    2. Nullable database columns — project: str | None, completed_at: str | None
    3. Optional filters — status: list[str] | None = None
    4. Record-not-found semantics — get_task() -> TaskDict | None
    5. Transaction composition — conn: sqlite3.Connection | None

    All trace back to: - Nullable SQL columns (confirmed via migration files) - Optional function parameters with = None defaults - Return values that legitimately can be None

    Critical Issues from Codebase Audit¶

    Fixed in PR #594¶

    • ✅ Transaction isolation: save_document() now commits after tags
    • ✅ Null-safety: 10+ locations fixed with null checks
    • ✅ Validation: Pagination and date parameters validated
    • ✅ SQL injection: Column name whitelists prevent dynamic SQL vulnerabilities

    Remaining Work¶

    • ⚠️ connection.py:53-56 — Connection closed without rollback on exception
    • ⚠️ cascade.py:257-308 — Opens two separate connections for related operations
    • ⚠️ cascade.py:67-87 — Non-atomic check-then-act in update_cascade_stage()
    • ⚠️ Migration error handling — Schema version set even if migration fails

    Missing Error Handling in Migrations¶

    migrations.py:2217-2240 sets schema version even if migration partially fails:

    try:
    for version, description, migration_func in MIGRATIONS:
        if version > current_version:
            migration_func(conn)  # ❌ Exception lost
            set_schema_version(conn, version)  # ❌ Version incremented anyway
finally:
    conn.close()

    Impact: Corrupted schema state; future migrations skipped incorrectly.

    Fix:

    try:
    for version, description, migration_func in MIGRATIONS:
        if version > current_version:
            try:
                migration_func(conn)
                set_schema_version(conn, version)
            except Exception as e:
                conn.rollback()
                raise RuntimeError(f"Migration {version} failed") from e
finally:
    conn.close()

    Common Patterns and Anti-Patterns¶

    ✅ Good Patterns¶

    • Parameterized SQL for all user-facing queries
    • FTS5 query escaping in search.py
    • No shell=True subprocess calls
    • API keys from environment variables
    • TypedDict for all known dict shapes
    • Whitelist validation for dynamic column names

    ❌ Anti-Patterns to Avoid¶

    • Unchecked fetchone()[0]: Always check for None before indexing (fixed in PR #594)
    • Silent exception swallowing: except Exception: pass appears in activity_items.py, ui_config.py, cascade_service.py
    • Dynamic SQL column names without whitelist: Fixed in PR #594
    • Querying FTS5 virtual table directly: Must JOIN on rowid
