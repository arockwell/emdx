    EMDX Implementation Planning: Best Practices Guide¶

    Overview¶

    This guide documents best practices for planning and implementing features in EMDX, derived from successful patterns across multiple feature implementations. It covers testing strategies, implementation planning, and interactive development workflows.

    Key Concepts¶

    Implementation Strategy¶

    Phased Development Successful implementations follow a multi-phase approach: - Diagnostic Phase: Understand the current system before making changes - Detection Phase: Identify patterns and requirements - Implementation Phase: Build the solution incrementally - Validation Phase: Test and verify functionality

    Example: The Claude-Save Hook implementation demonstrates this pattern perfectly - starting with command logging to understand the actual format, then building detection logic, and finally implementing the fix.

    Testing Plans¶

    Comprehensive Test Coverage Testing plans should address multiple dimensions: - Functionality Tests: Core feature works as expected - Edge Cases: Special characters, boundary conditions, error states - Integration Tests: Feature works with existing system - Performance Tests: Acceptable speed with realistic data volumes - User Experience Tests: Intuitive and discoverable

    Example Test Checklist:

    - [ ] Feature works with test data
- [ ] Special characters handled correctly
- [ ] Error messages are clear
- [ ] Performance acceptable with large datasets
- [ ] UI/UX is intuitive
- [ ] Documentation updated

    Interactive Development¶

    Command-Line First Design EMDX follows a "same commands, different output modes" philosophy: - Default output: Plain text (machine/pipe friendly) - --rich flag: Colored Rich output with visual enhancements - --json flag: Structured machine output - TUI: Interactive visual interface

    All features should work across these modes with consistent behavior.

    Architecture & Design Decisions¶

    Modal Interface Pattern¶

    Problem: Key binding conflicts between navigation (j/k) and actions in interactive browsers

    Solution: Vim-like modal interface with distinct NORMAL and ACTION modes

    mode_bindings = {
    "normal": {
        "j": "down",
        "k": "up", 
        "g": "first",
        "G": "last",
        "/": "toggle-search",
        "a": "switch-to-action-mode",
        "enter": "execute(view)"
    },
    "action": {
        "e": "execute(edit)+reload",
        "d": "execute(delete)+reload",
        "v": "execute(view)",
        "r": "execute(rename)+reload",
        "t": "execute(tag)+reload",
        "esc": "switch-to-normal-mode"
    }
}

    Benefits: - No key conflicts between modes - Vim-like feel for experienced users - Easy to extend with new actions - Clear visual feedback via prompt changes

    Safety-First Delete Operations¶

    Mandatory Safety Features: - Confirmation prompts by default - Show full document preview before deletion - Soft delete option (mark as deleted but preserve data) - Undo capability for recently deleted items

    function claude-delete
    # Show what will be deleted
    claude-view $id

    # Confirmation prompt (unless --force)
    read -P "Delete this document? [y/N] " confirm

    # Perform deletion
    if test "$confirm" = "y"
        _clauding_db "UPDATE knowledge SET deleted_at = CURRENT_TIMESTAMP WHERE id = $id"
    end
end

    Document Relationships¶

    Link Syntax Support: - [[Document Title]] - Wiki-style links - #2106 - Reference by ID - {{fuzzy match}} - Fuzzy match link - See also: #2106, #2107 - Multiple references

    Bidirectional Tracking:

    CREATE TABLE knowledge_links (
    source_id INTEGER REFERENCES knowledge(id),
    target_id INTEGER REFERENCES knowledge(id),
    link_type TEXT DEFAULT 'reference'
);

    Automatically maintain both references and referenced_by arrays for quick navigation.

    Version History Strategy¶

    Git-Style Versioning: - Automatic version creation on content changes - Store diffs for minor changes (space efficient) - Compress old versions - Smart change summaries ("Added section: Implementation Details")

    CREATE TABLE knowledge_versions (
    id SERIAL PRIMARY KEY,
    document_id INTEGER REFERENCES knowledge(id),
    version_number INTEGER NOT NULL,
    content TEXT,
    change_summary TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

    Status Line Design¶

    Command Help + Context Display:

    Bottom status bar should show both available commands and current state:

    [dim]j/k:nav /:search e:edit d:delete v:view r:refresh q:quit[/dim] | [bright]45/150 docs[/bright]

    Mode-Aware Display: - NORMAL mode: Show navigation and action commands - SEARCH mode: Show search-specific commands and filter status - Use rich markup for visual hierarchy

    Sub-Agent Parallelization¶

    Problem: Sequential operations are slow for financial data retrieval

    Solution: Parallel sub-agents for independent tasks

    # Sequential (slow): 8 seconds total
emails = []
emails += search_emails("bank statements")  # 2s
emails += search_emails("credit cards")     # 2s
emails += search_emails("investments")      # 2s
emails += search_emails("bills")           # 2s

# Parallel (fast): 2 seconds total
sub_agents = [
    EmailSubAgent("bank statements"),
    EmailSubAgent("credit cards"),
    EmailSubAgent("investments"),
    EmailSubAgent("bills")
]
results = parallel_execute(sub_agents)

    Best Practices: - Define clear boundaries for each sub-agent - Minimize dependencies between sub-agents - Use appropriate granularity (don't create too many small sub-agents) - Handle failures gracefully - Aggregate results intelligently

    Gmail Search Optimization¶

    Comprehensive Query Pattern:

    # Instead of multiple searches, use one comprehensive query
after:2025/06/01 before:2025/07/01 (bank OR credit OR payment OR invoice OR bill OR statement)

    Category-Based Searches:

    # Retail
from:(amazon.com OR ebay.com OR target.com)

# Services  
from:(netflix.com OR spotify.com OR adobe.com)

# Financial
from:(*bank.com OR *card.com) subject:(statement OR payment)

    Email Management¶

    Structured Analysis Approach: 1. Use comprehensive search queries to minimize API calls 2. Categorize emails (Retail, Services, Financial, etc.) 3. Extract key information (balances, due dates, transactions) 4. Generate weekly reports with actionable insights

    Security Best Practices: - Enable 2FA on email accounts - Never store full account numbers in plain text - Review account access regularly - Watch for phishing attempts with targeted searches

    Document Organization¶

    Tags vs. Task Categories: - Documents: Use plain text tags (gameplan, active, analysis) - Tasks: Use categories (FEAT, FIX, ARCH) and epics

    Link Graph: - Build connections between related documents - Automatic backlink tracking - Visual graph representation - "See Also" suggestions based on shared tags and similar content

    Configuration Management¶

    Hook-Based Workflow:

    {
  "hooks": {
    "before_tool_use": "fish -c 'source ~/.config/fish/functions/hook.fish && hook'"
  }
}

    Testing Workflow: 1. Create hook script 2. Configure Claude Code to use it 3. Test with simple commands 4. Monitor logs for verification 5. Iterate based on usage
