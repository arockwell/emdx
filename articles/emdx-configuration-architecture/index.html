    emdx Configuration Architecture¶

    Overview¶

    The emdx project uses a centralized configuration system located in emdx/config/ that manages models, timeouts, environment variables, and CLI tool settings. This configuration layer was consolidated in February 2026 as part of a technical debt audit (DEBT-7) to eliminate scattered hardcoded values across the codebase.

    Core Configuration Modules¶

    emdx/config/constants.py — Central Constants¶

    The single source of truth for all configurable values across the project.

    Timeout Constants:

    Constant Value Purpose
    DEFAULT_TIMEOUT_SECONDS / DEFAULT_CLI_TIMEOUT_SECONDS / CLI_SYNC_DEFAULT_TIMEOUT 300s (5 min) Standard task timeout
    DELEGATE_TIMEOUT_SECONDS 600s (10 min) Delegate command execution
    CASCADE_STAGE_TIMEOUT / CASCADE_TIMEOUT_SECONDS / STAGE_DEFAULT_TIMEOUT 300s (5 min) Analysis/planning stages
    CASCADE_IMPLEMENTATION_TIMEOUT / IMPLEMENTATION_TIMEOUT_SECONDS / STAGE_IMPLEMENTATION_TIMEOUT 1800s (30 min) Code generation tasks
    DISCOVERY_TIMEOUT_SECONDS / DISCOVERY_COMMAND_TIMEOUT 30s Shell discovery commands (--each)
    CLI_VERSION_CHECK_TIMEOUT / SUBPROCESS_SHORT_TIMEOUT_SECONDS / VERSION_CHECK_TIMEOUT_SECONDS 5s Version/status checks
    CLI_AUTH_CHECK_TIMEOUT / SUBPROCESS_MEDIUM_TIMEOUT_SECONDS / AUTH_CHECK_TIMEOUT_SECONDS 10s Authentication verification
    GITHUB_AUTH_TIMEOUT 10s gh auth token retrieval
    PROCESS_TERMINATE_TIMEOUT 3s Graceful termination wait
    PROCESS_KILL_TIMEOUT 1s Force kill wait
    STALE_EXECUTION_TIMEOUT_SECONDS (implementation-defined) Stale execution detection

    UI Debounce Constants:

    Constant Value Purpose
    DEBOUNCE_FTS_MS 300ms Full-text search debounce
    DEBOUNCE_TAGS_MS 300ms Tag search debounce
    DEBOUNCE_SEMANTIC_MS 500ms Semantic search debounce
    DEBOUNCE_COMBINED_MS 400ms Combined search debounce

    Health/Quality Constants:

    • HEALTH_SCORE_GOOD_THRESHOLD = 80 — Score threshold for "good" quality
    • HEALTH_SCORE_WARNING_THRESHOLD = 60 — Score threshold for warnings

    ML/Search Constants:

    • MIN_EMBEDDINGS_FOR_SEMANTIC = 50 — Minimum embeddings required for semantic search
    • DEFAULT_MAX_TOKENS = 1000 — Maximum tokens for LLM API calls

    Environment Variable Names:

    All environment variable names are exported as constants with ENV_ prefix:

    • ENV_CLI_TOOL = "EMDX_CLI_TOOL" — CLI tool selection
    • ENV_SAFE_MODE = "EMDX_SAFE_MODE" — Safe mode toggle
    • ENV_TEST_DB = "EMDX_TEST_DB" — Test database path
    • ENV_CODE_THEME = "EMDX_CODE_THEME" — Code syntax theme
    • ENV_ANTHROPIC_API_KEY — Anthropic API key (sensitive)
    • ENV_CURSOR_API_KEY — (Legacy, removed in 2026-02)
    • ENV_GITHUB_TOKEN — GitHub token (sensitive)
    • ENV_EMDX_DATABASE_URL — Database connection URL

    Environment Variable Validation:

    • BOOL_TRUTHY_VALUES — Set of accepted boolean-like values for env var parsing
    • ENV_VARS — Dictionary documenting all known environment variables

    Miscellaneous:

    • DEFAULT_EDITOR = "nano" — Fallback editor if EDITOR not set

    Architecture & Design Decisions¶

    Centralization Rationale (DEBT-7 Audit, 2026-02-14)¶

    Problem: Configuration debt was identified across 13+ files: - Model names hardcoded in 5 files - Timeout values scattered across 10+ files - Environment variables read without validation - No central documentation of configurable values

    Solution: Three-phase centralization:

    1. Immediate (PR #511, #526):
    2. Created config/models.py as single source of truth for model names
    3. Moved all timeout values to config/constants.py with TIMEOUT_* prefix

    4. Added ENV_VAR_DEFINITIONS dictionary with metadata

    5. Short-term:

    6. Added environment variable validation helpers in config/settings.py
    7. Defined exit code constants (EXIT_TIMEOUT = 124, etc.)

    8. Created config/ui_layout.py for UI dimension constants (planned)

    9. Long-term:

    10. YAML config file support for user-adjustable settings (planned)
    11. emdx config command for runtime configuration (planned)
    12. Support reading defaults from ~/.config/emdx/defaults.yaml (planned)

    Benefits Achieved: - Single source of truth for all configuration values - Easier maintenance (change one constant, update entire codebase) - Better IDE support (autocomplete, refactoring) - Environment validation catches invalid configuration early - Reduced duplication (model strings defined once, used everywhere)

    Timeout Strategy¶

    The codebase uses task-based timeouts rather than global defaults:

    • Standard tasks (5 min): Searching, analyzing, reading files
    • Code generation tasks (30 min): Writing code, refactoring, synthesis
    • Delegate tasks (10 min): Parallel research, investigations
    • Subprocess checks (5-10s): Version checks, auth validation

    Timeouts are configurable via constants rather than CLI flags, allowing users to adjust globally for slow networks or large operations.

    Model Selection Hierarchy¶

    Model resolution follows this precedence:

    1. Explicit parameter (--model opus)
    2. CLI tool default (from CLI_CONFIGS[tool].default_model)
    3. Global default (DEFAULT_CLAUDE_MODEL)

    The resolve_model() helper handles alias expansion ("opus" → "claude-opus-4-6").

    Environment Variable Patterns¶

    Boolean Variables:

    # Correct pattern (uses BOOL_TRUTHY_VALUES)
safe_mode = os.environ.get("EMDX_SAFE_MODE", "0").lower() in BOOL_TRUTHY_VALUES

    API Keys (Sensitive):

    # Use constant + validation
[REDACTED] validate=True)

    Optional with Fallback:

    # Use centralized getter with default
cli_tool = get_default_cli_tool()  # Checks ENV_CLI_TOOL or defaults to "claude"

    Type Annotations for Configuration¶

    Configuration dataclasses use optional types judiciously:

    Good Pattern (Legitimate Optionals):

    @dataclass
class ExecutionConfig:
    doc_id: int | None = None        # Document may not exist
    model: str | None = None          # Falls back to default
    timeout: int | None = None        # Uses CLI default if None

    Avoid (Unnecessary Optionals):

    # Don't do this if the value is always set
exit_code: int | None = None  # Should be `int = 0` if always assigned

    See Document #6492 (None Type Analysis) for detailed analysis of | None annotations in executor files.
