    claude-config.fish: __claude_psql and root cause¶

    Overview¶

    The claude-config.fish file is a core component of the clauding knowledge base system that provides centralized database configuration and a critical helper function (__claude_psql) used throughout the codebase. Despite its importance, improper sourcing order has historically caused "Unknown command" errors that confuse users and break the entire system.

    Key Concepts¶

    __claude_psql Helper Function¶

    The __claude_psql function is a database connection wrapper that:

    • Eliminates hardcoded database credentials scattered throughout the codebase
    • Uses environment variables for configuration (PGUSER, PGDATABASE, etc.)
    • Provides a consistent interface for all PostgreSQL operations
    • Enables the security layer (__claude_safe_insert) to function properly

    Sourcing Order Dependency¶

    Fish functions have an explicit dependency chain that must be respected:

    1. claude-config.fish - Defines __claude_psql and environment variables
    2. claude-security.fish - Calls __claude_psql for safe SQL operations
    3. claude-functions.fish - Sources claude-security.fish and uses both helpers

    If claude-config.fish is not sourced first, any function that needs __claude_psql fails with:

    __claude_psql: Unknown command

    Root Cause Analysis¶

    The Core Problem¶

    The claude-save command and other claude functions fail with psql parameter errors when called through the Bash tool. The specific error pattern:

    psql: warning: extra command-line argument "title=..." ignored
ERROR:  syntax error at or near "-"
LINE 1: -v
        ^
ERROR:  syntax error at or near ":"
LINE 3:         VALUES (:'title', :'content', :'project',
                        ^

    This error stems from two separate issues:

    1. Missing __claude_psql function - The helper isn't loaded because claude-config.fish wasn't sourced
    2. Shell escaping issues - When commands pass through multiple shell layers (Bash tool fish psql), quotes and special characters get mangled

    Why It Happens¶

    The installation process creates ~/.config/fish/conf.d/claude.fish with environment variables but does not source the function definitions from claude-config.fish. Users must manually add sourcing commands to their fish config, but:

    • The documentation mentions this in "Permanent Setup" but it's not prominent
    • Error messages don't indicate the missing dependency
    • The installation doesn't validate that sourcing is configured correctly

    Dependency Chain¶

    claude-functions.fish
   sources claude-security.fish dynamically
       calls __claude_psql function
           defined in claude-config.fish (NOT SOURCED!)

    Impact¶

    • Users get cryptic errors when trying to use claude commands
    • Natural workflow of saving documents is broken
    • Forces use of lower-level MCP PostgreSQL tools
    • Creates confusion about whether the system is properly installed

    Architecture & Design Decisions¶

    Security Improvements (Plan #2081)¶

    The clauding project implemented proper security fixes that depend on claude-config.fish:

    • claude-security.fish module uses PostgreSQL's \set commands with parameterized queries
    • No hardcoded credentials in main functions (environment variables only)
    • __claude_safe_insert function prevents SQL injection

    However, completion files still contain hardcoded credentials (psql -U alexrockwell -d claudedb), creating a security inconsistency where the main system is secure but completions are not.

    Configuration System Design¶

    claude-config.fish centralizes:

    • Database connection parameters (username, database, host, port)
    • __claude_psql wrapper for consistent database access
    • Environment variable configuration

    This design requires that claude-config.fish be sourced before any other claude module.

    Multiple Function Versions Problem¶

    The codebase contains duplicate implementations:

    • claude-functions.fish
    • claude-functions-enhanced.fish
    • claude-functions-scale.fish

    These variants all expect __claude_psql to exist, amplifying the sourcing order issue across the codebase.

    Implementation Details¶

    Correct Sourcing Pattern¶

    To use clauding commands, source files in this exact order:

    source ~/dev/worktrees/clauding-main/core/claude-config.fish
source ~/dev/worktrees/clauding-main/core/claude-functions.fish
source ~/dev/worktrees/clauding-main/core/claude-direct-fixed.fish

    Proposed Solutions¶

    Three approaches were identified to fix the sourcing issue:

    Option A: Auto-source in claude-functions.fish

    # At top of claude-functions.fish
if not functions -q __claude_psql
    source (dirname (status -f))/claude-config.fish
end

    Option B: Single entry point

    Create claude-init.fish:

    #!/usr/bin/env fish
set -l claude_dir (dirname (status -f))
source $claude_dir/claude-config.fish
source $claude_dir/claude-functions.fish
source $claude_dir/claude-direct-fixed.fish
echo " Claude knowledge base ready!"

    Option C: Update installer

    Modify install-claude.fish to write complete sourcing commands to the user's fish config automatically.

    Error Detection¶

    A proposed improvement is to add runtime checks:

    # In claude-functions.fish
if not functions -q __claude_psql
    echo "ERROR: claude-config.fish not loaded. Please source it first." >&2
    return 1
end

    This provides a helpful error message instead of "Unknown command."

    Diagnostic Command¶

    A claude-check-setup command was proposed to diagnose configuration issues:

    function claude-check-setup
    if functions -q __claude_psql
        echo " __claude_psql found"
    else
        echo " __claude_psql missing - source claude-config.fish"
    end

    if test -n "$PGUSER"
        echo " Database user configured: $PGUSER"
    else
        echo "  PGUSER not set"
    end

    # Additional checks...
end

    Related Topics¶

    Claude-list Simplification¶

    The claude-list command also suffered from sourcing issues. It used hardcoded credentials (psql -U alexrockwell) instead of the __claude_psql helper, preventing it from working on other machines.

    The fix involved:

    1. Replacing hardcoded psql calls with __claude_psql
    2. Adding pagination and filtering options
    3. Implementing proper error handling

    Completion Files Cleanup¶

    The completion files in completions/ directory still contain 13 instances of hardcoded database credentials. These need to be updated to use __claude_psql for consistency and security.

    MCP PostgreSQL Workaround¶

    When claude-save fails due to sourcing issues, users can work around it by using the MCP PostgreSQL tool directly:

    mcp__postgres-pro__execute_sql "INSERT INTO claude.knowledge ..."

    This bypasses shell escaping issues but defeats the purpose of the claude command interface.
