    input_tokens / output_tokens / tokens_used¶

    Overview¶

    The emdx project tracks token usage and API costs for delegate executions at the database level. Three key metrics are stored in the executions table: input_tokens, output_tokens, and tokens_used (total tokens). These fields enable cost tracking, usage analytics, and execution monitoring across the system.

    Key insight: Token data is only written after execution completes all three fields remain at their default values (0) while a delegate is running.

    Key Concepts¶

    Token Metrics Definition¶

    Metric Type Purpose
    input_tokens INTEGER Number of tokens sent to the Claude API (prompt + context)
    output_tokens INTEGER Number of tokens generated by Claude (response)
    tokens_used INTEGER Total tokens consumed (input_tokens + output_tokens)
    cost_usd REAL Actual cost in USD based on model pricing

    These metrics are central to understanding delegate resource consumption and providing transparency around API costs.

    Data Lifecycle¶

    1. Execution created  cost/token fields set to 0
2. Delegate runs  fields remain 0 (no live updates)
3. Delegate completes  unified_executor.py writes all metrics in one atomic update
4. Dashboard/status displays completed execution with full cost data

    Architecture & Design Decisions¶

    Schema Evolution¶

    The token tracking fields were added in migration 036 (migration_036_add_execution_metrics):

    # Migration 036 added to executions table:
task_id INTEGER REFERENCES tasks(id)
cost_usd REAL DEFAULT 0.0
tokens_used INTEGER DEFAULT 0
input_tokens INTEGER DEFAULT 0
output_tokens INTEGER DEFAULT 0

    This migration was part of the broader execution metrics tracking initiative that also linked executions back to tasks.

    Schema-Model Synchronization Issues¶

    Historical bug (MEDIUM severity): The Execution dataclass was initially missing these fields after migration 036 was applied, causing a schema-model mismatch. This was identified in multiple technical debt audits:

    • Functions could write the data via dynamic update_execution() but couldn't read it back
    • The dataclass always returned default values (0) even when database had real data
    • Fixed in PR #487, PR #509, PR #561 by adding the 6 missing fields to the dataclass

    Data Availability Constraint¶

    Token data is only available post-completion because: - The Claude API returns usage data in the final response - Streaming responses don't provide incremental token counts - Cost calculation requires knowing the final model and token counts

    Implication for dashboards: A "running delegates" view cannot show accurate cost/token data those cells must remain empty or show "" until completion.

    Single-Write Pattern¶

    All token metrics are written in a single atomic update in unified_executor.py (lines 474-487):

    update_execution(
    exec_id,
    cost_usd=result.cost_usd,
    tokens_used=result.tokens_used,
    input_tokens=result.input_tokens,
    output_tokens=result.output_tokens,
)

    This guarantees consistency you never have a partial state where input_tokens is set but output_tokens is missing.

    Type Safety Journey¶

    The project underwent multiple iterations to achieve type safety for execution results:

    1. Initial state: ExecutionResult.to_dict() returned dict[str, Any]
    2. Type safety audit (2026-02-15): Identified this as an actionable improvement
    3. PR #696, #697: Added ExecutionResultDict TypedDict to emdx/services/types.py
    4. Duplicate PR incident: Both #697 and #698 merged, causing mypy error: no-redef on main
    5. Resolution: Deduplication and standardization on ExecutionResultDict

    Current definition in emdx/services/types.py:

    class ExecutionResultDict(TypedDict):
    """Serialized execution result from UnifiedExecutor."""
    success: bool
    execution_id: int
    log_file: str
    output_doc_id: int | None
    output_content: str | None
    tokens_used: int
    input_tokens: int
    output_tokens: int
    cost_usd: float
    execution_time_ms: int
    error_message: str | None
    exit_code: int | None
    cli_tool: str

    Implementation Details¶

    Execution Model¶

    Located in emdx/models/executions.py, the dataclass now correctly includes all token fields:

    @dataclass
class Execution:
    id: int
    doc_id: Optional[int]
    doc_title: str
    status: str
    started_at: datetime
    completed_at: Optional[datetime]
    log_file: str
    exit_code: Optional[int]
    working_dir: Optional[str]
    pid: Optional[int]
    cascade_run_id: Optional[int] = None
    task_id: Optional[int] = None
    cost_usd: float = 0.0
    tokens_used: int = 0
    input_tokens: int = 0
    output_tokens: int = 0

    Query Pattern¶

    All functions that query executions use a shared column list _EXECUTION_COLUMNS (refactored for consistency):

    _EXECUTION_COLUMNS = """
    id, doc_id, doc_title, status, started_at, completed_at,
    log_file, exit_code, working_dir, pid, cascade_run_id,
    task_id, cost_usd, tokens_used, input_tokens, output_tokens
"""

    Helper function _row_to_execution(row) parses SQLite rows with backward compatibility for legacy DBs missing the newer columns.

    Functions using this pattern: - get_execution(exec_id) Fetch single execution by ID - get_recent_executions(limit) Recent executions ordered by start time - get_running_executions() Active delegates (status='running') - get_stale_executions(timeout_minutes) Delegates exceeding timeout threshold

    Status Display¶

    The emdx exec health command provides the richest view of running delegates:

    Column Source
    ID executions.id
    Title executions.doc_title
    PID executions.pid
    CPU/MEM psutil.Process(pid) (live)
    Runtime now - executions.started_at (computed)
    Health Execution.is_zombie property (PID check via os.kill(pid, 0))

    The emdx status command currently uses the tasks table, not executions, to show active delegates.

    Cost Estimation¶

    The synthesis_service.py provides pre-execution cost estimates via estimate_cost():

    def estimate_cost(self, doc_ids: list[int]) -> dict[str, float | int]:
    # Returns:
    # {
    #   "input_tokens": int,
    #   "output_tokens": int,
    #   "estimated_cost": float
    # }

    Type safety note: The return type uses dict[str, float | int] instead of a TypedDict. A CostEstimate TypedDict was proposed in the 2026-02-15 audit but not yet implemented:

    class CostEstimate(TypedDict):
    input_tokens: int
    output_tokens: int
    estimated_cost: float

    Related Topics¶

    Database Foreign Keys¶

    The task_id column added in migration 036 links executions back to the task system. This enables: - Grouping executions by task in the Activity screen - Tracking task-level cumulative costs - Linking delegate output documents to their creating tasks

    Execution Heartbeat (Deprecated)¶

    Earlier versions tracked liveness via a last_heartbeat column (migration 005), but this was dropped in migration 013 when the executions table was recreated. Functions like update_execution_heartbeat() and get_stale_executions() originally relied on this column but now use PID-based detection instead.

    The heartbeat functionality was restored in migration 037 (PR #509) but is no longer actively used PID checks via Execution.is_zombie are more reliable.

    Migration Integrity¶

    Migration 038 fixed orphan-prone foreign keys on related tables (gists, gdocs, export_history, cascade_runs) by adding ON DELETE CASCADE or ON DELETE SET NULL behavior. This prevents orphaned execution records when parent documents or tasks are deleted.

    Type Safety Audit Context¶

    The token tracking fields were part of a broader type safety initiative in February 2026:

    • Phase 1: Identified 8+ functions returning dict[str, Any] with known shapes
    • Phase 2: Created TypedDicts in emdx/services/types.py for structured data
    • Phase 3: Fixed schema-model mismatches across executions, tasks, and cascade systems

    The ExecutionResultDict TypedDict was one outcome, ensuring compile-time type checking for all execution result consumers.
