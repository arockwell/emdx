    emdx prime and Smart Context Features¶

    Overview¶

    emdx prime is the session initialization command that provides Claude Code agents with relevant work context at the start of a session. It answers the question "what should I work on?" by showing ready tasks, in-progress work, git state, and recent failures.

    The command has evolved through multiple phases, incorporating "smart priming" features that provide context-aware information to help agents understand project state, recent activity, and what needs attention.

    Core Features¶

    Basic Prime Output¶

    The default emdx prime output includes:

    • Project identification Detected from git repository name
    • Usage instructions Common commands for save, search, view, tags, and tasks
    • Active epics Parent epics with progress indicators
    • Ready tasks Unblocked tasks available to work on
    • In-progress tasks Currently active work
    • Cascade queue status Background analysis pipeline state
    • Git context Current branch, recent commits, open PRs
    • Key documents Most-viewed documents by access count
    • Stale documents Important docs that haven't been viewed recently
    • Recent failures Failed delegate executions from the last 24 hours

    Output Modes¶

    emdx prime                    # Text output (default)
emdx prime --format json      # JSON output
emdx prime --verbose          # Include extra context (key docs, stale docs)
emdx prime --quiet            # Minimal output (omit failures)

    Smart Priming Features¶

    Git Context (_get_git_context())¶

    Location: emdx/commands/prime.py

    Captures current git state to provide situational awareness:

    • Current branch Via git branch --show-current
    • Recent commits Last 3 commits via git log --oneline -3
    • Open PRs Via gh pr list --state=open --limit=5 --json number,title,headRefName

    Graceful degradation: - Handles git not installed (FileNotFoundError) - Handles git timeout (subprocess.TimeoutExpired) - Handles not a git repo (non-zero return code) - Continues without PR data if gh CLI not installed

    Implementation notes: - Uses subprocess.run with capture_output=True and timeout - Parses JSON output from gh CLI - Integrated into both text and JSON output modes

    Key Documents (_get_key_docs())¶

    Location: emdx/commands/prime.py

    Shows most-viewed documents to surface important reference material:

    SELECT id, title, access_count 
FROM documents 
WHERE deleted_at IS NULL AND access_count > 0 
ORDER BY access_count DESC 
LIMIT ?

    Display format: #ID "Title" N views

    Only shown in: --verbose mode

    Recent Activity and Staleness Detection¶

    Recent activity uses the last_accessed_at column to show documents accessed in the last 7 days.

    Staleness detection is handled by the dedicated stale.py module with importance scoring: - Uses StalenessLevel enum (CRITICAL/WARNING/INFO) - Importance scoring considers tag weights (security=3, gameplan=2) - get_top_stale_for_priming() is the integration point - See emdx stale list for standalone command

    Recent Failures¶

    Location: emdx/commands/prime.py (lines 495-564)

    Shows failed delegate executions from the last 24 hours with retry commands.

    Query logic:

    SELECT * FROM executions
WHERE status = 'failed'
  AND completed_at > datetime('now', '-24 hours')
ORDER BY completed_at DESC
LIMIT 5

    Error snippets are extracted from log files using _extract_error_snippet(): - Reads the last N lines from ~/.config/emdx/logs/<exec_id>.log - Returns None for missing/empty files - Uses Path.expanduser() for tilde expansion

    Text output format:

    RECENT FAILURES (2):

  #101  Auth module crash (exit 1)
         Error: connection refused
         Retrying...
         Retry: emdx delegate --retry 101
  #102  Database timeout (exit 127)
         Retry: emdx delegate --retry 102

    JSON output format:

    {
  "recent_failures": [
    {
      "id": 101,
      "title": "Auth module crash",
      "completed_at": "2024-01-15T10:00:00",
      "exit_code": 1,
      "task_id": 42,
      "error_snippet": "Error: connection refused\nRetrying...",
      "retry_command": "emdx delegate --retry 101"
    }
  ]
}

    Constants: - FAILURE_LOOKBACK_HOURS = 24 How far back to search - MAX_RECENT_FAILURES = 5 Maximum to display - ERROR_SNIPPET_LINES = 5 Lines to extract from logs

    Respects --quiet flag Recent failures hidden in quiet mode.

    Enhanced Ask Features¶

    The emdx ai ask and emdx ai context commands were enhanced with filtering and enhanced output features (PR #453 / FEAT-7).

    Tag and Recent Filters (_add_filters())¶

    Location: emdx/services/ask_service.py

    CLI usage:

    # Filter by tags
emdx ai ask "What are our security patterns?" --tags "security,active"

# Filter by recency
emdx ai ask "What changed recently?" --recent 7

# Combined filters
emdx ai ask "What's the auth strategy?" --tags "gameplan" --recent 30

# Context retrieval with filters (for piping to Claude CLI)
emdx ai context "security best practices" --tags "security" | claude

    Implementation: - Text aliases auto-expand to emojis via normalize_tag_to_emoji() - Filters applied to both keyword and semantic retrieval paths - _get_filtered_doc_ids() helper joins document_tags and tags tables - Recent filter: created_at > datetime('now', '-N days')

    Enhanced Answer Output¶

    Location: emdx/services/ask_service.py

    The Answer dataclass was enhanced with:

    1. Source titles source_titles: list[tuple[int, str]]
    2. Display format: #ID "Title" instead of bare #ID

    3. Populated during retrieval

    4. Confidence indicator confidence: str

    5. "high" for 3+ sources
    6. "medium" for 1-2 sources
    7. "low" for 0 sources

    8. Display with color in Rich output (green/yellow/red panel borders)

    9. Context budget ~3000 tokens cap

    10. Constant: CONTEXT_BUDGET_CHARS = 12000
    11. Tracks total chars across all context docs
    12. Per-doc cap of 3000 chars or remaining budget
    13. Breaks when budget exceeded

    Panel display format:

     Answer [HIGH confidence] 
 The auth module uses JWT tokens with RSA signing.

    Architecture & Design Decisions¶

    Why Tasks Table for Failures (Not Executions)¶

    Initial proposal (Document #6542): Use the tasks table with get_failed_tasks() because: - Tasks have an error field that stores error messages - status.py already implements _show_failed_tasks() using this function - Tasks have prompt field needed for retry commands - Executions don't have error messages stored; errors live in log files

    Final implementation (Document #6560): Uses the executions table instead because: - More accurate source of truth for delegate execution failures - completed_at timestamp shows when the failure occurred - exit_code provides diagnostic information - Error snippets extracted from log files (last N lines) - Better handles edge cases (missing prompts, null exit codes)

    Filter Reusability¶

    The _add_filters() helper in ask_service.py was extracted to avoid duplication between _retrieve_keyword() and _retrieve_semantic(). This followed code review feedback on PR #453.

    Before: Filter logic duplicated in both methods After: Single module-level helper handles all filter combinations

    Column Name Consistency¶

    Early implementation of smart priming had a column name mismatch (accessed_at vs last_accessed_at). The current codebase consistently uses last_accessed_at for document access tracking.

    Implementation Details¶

    Subprocess Handling for Git Commands¶

    Git context uses subprocess.run with proper error handling:

    def _get_git_context() -> dict[str, Any]:
    try:
        result = subprocess.run(
            ["git", "branch", "--show-current"],
            capture_output=True,
            text=True,
            timeout=5,
            check=True
        )
        branch = result.stdout.strip()
    except (FileNotFoundError, subprocess.TimeoutExpired, subprocess.CalledProcessError):
        return {"error": "git not available"}

    Hook Integration¶

    Claude Code hooks in .claude/settings.json handle session lifecycle:

    Hook Event What it does
    prime.sh SessionStart Injects KB context (ready tasks, in-progress)
    save-output.sh Stop Auto-saves output to KB (delegate sessions only)
    session-end.sh SessionEnd Updates task status (delegate sessions only)

    Env vars recognized by hooks:

    Variable Set by Used by Purpose
    EMDX_AUTO_SAVE delegate save-output.sh Enable auto-save ("1" to activate)
    EMDX_DOC_ID delegate prime.sh Include document as context
    EMDX_TASK_ID delegate prime.sh, session-end.sh Track task lifecycle

    Token Efficiency¶

    Smart priming targets <500 tokens for compact output: - Truncates titles to 40 characters - Truncates error snippets to 60 characters (first line only) - Limits to 5 items per section (ready tasks, failures, key docs) - Uses compact formatting (no extra whitespace)

    Test Coverage¶

    Location: tests/test_commands_prime.py and tests/test_ask.py

    Git context tests: - Mock subprocess calls for branch/commits/PRs - Test graceful degradation (git failure, gh CLI missing, timeout) - Test JSON output validity

    Recent failures tests: - Test error snippet extraction (missing file, empty file, whitespace-only, tilde expansion) - Test quiet mode (failures hidden) - Test long title truncation - Test null exit code handling

    Ask feature tests: - Tag filtering query structure validation - Recent days filtering (1, 7, 30, 90, 365 days) - Confidence level calculation (high/medium/low) - Machine output format validation - Filter application in keyword and semantic retrieval

    Related Topics¶

    Delegate Workflow Integration¶

    Prime output is consumed by Claude Code agents at session start via the prime.sh hook. For delegate sessions: - EMDX_DOC_ID injects specific document context - EMDX_TASK_ID tracks task lifecycle - Recent failures provide retry commands for failed attempts

    Knowledge Base Organization¶

    Prime leverages multiple KB organization mechanisms: - Tasks Epics, categories, dependencies - Tags Plain text classification (gameplan, active, security) - Access tracking last_accessed_at, access_count for staleness/importance

    Cascade Queue¶

    The cascade queue runs background analysis to auto-link documents and extract entities. Prime shows queue status to indicate when analysis is pending vs complete.
