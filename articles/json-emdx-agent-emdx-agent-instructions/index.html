    --json: emdx agent and emdx agent instructions¶

    Overview¶

    EMDX operates as a dual-interface knowledge layer that serves both human developers and AI agents through different output modes. The --json flag represents the agent lens into the system, while rich tables and TUI provide the human interface. This design principlesame commands, different output modesallows EMDX to function as persistent memory for AI-assisted development.

    The system's identity has evolved from a personal knowledge base CLI into an agent-accessible knowledge layer where AI agents populate structured content that humans curate and direct across sessions.

    Key Concepts¶

    Dual-Interface Design¶

    EMDX serves two distinct user types with fundamentally different interaction patterns:

    Dimension Human Pattern Agent Pattern
    Content lifecycle Iterative, multi-edit Single creation, immutable
    Navigation Browse discover narrow Queue execute next
    Structure need Hierarchical (cognitive scaffolding) Flat (work queue)
    Output format Rich, visual, colored Structured, parseable
    Session model Exploratory, context-switching Linear task execution

    Output Modes¶

    EMDX implements four output modes to serve different use cases:

    Flag Purpose Use Case
    (default) Human-friendly Rich output Interactive terminal use
    --json Full structured output Agent consumption, scripting
    --quiet / -q Minimal output (just IDs) Piping to next command
    --porcelain Stable machine-parseable text Scripts needing stability guarantees

    The system uses layered detection to determine the appropriate mode: 1. Explicit flags (--json, --quiet) take precedence 2. Environment variables provide signals from agent runtime 3. TTY detection offers sensible defaults for non-interactive contexts

    The Delegate Pattern¶

    emdx delegate serves as the primary agent interface, acting as a bridge between Claude Code and the knowledge base:

    # Single task  results print AND persist
emdx delegate "audit auth for vulnerabilities"

# Parallel execution (up to 10 concurrent)
emdx delegate "check auth" "review tests" "scan for XSS"

# Synthesize results into coherent document
emdx delegate --synthesize "analyze auth" "analyze api" "analyze db"

# Feed document as context
emdx delegate --doc 47 "write an action plan from this analysis"

# Code changes with auto-PR
emdx delegate --pr "fix the auth bug"

    The delegate command: - Wraps Claude execution with KB persistence - Auto-creates tasks and tracks executions - Handles parallel, sequential, and discovery modes - Creates artifacts that persist beyond conversation

    Architecture & Design Decisions¶

    Strategic Positioning¶

    In a landscape where every major AI coding tool (Claude Code, Cursor, Windsurf, Devin) now has persistent memory, EMDX's differentiation centers on who controls it and how it's accessed:

    The Moat: 1. Tool-agnostic persistence: Works with any AI, not locked to specific vendors 2. Human-readable and searchable: Not opaque JSON buried in hidden directories 3. Dual-interface design: Same data, different lenses 4. Workflow-aware: Supports the agenthumanagent handoff pattern explicitly

    Why Not "Agent Workspace"?¶

    Critical analysis revealed that while usage patterns show heavy agent activity, positioning as an "agent workspace" would be strategically problematic:

    • Survivorship bias: Primary user is building agent features, skewing usage data
    • The KB IS the moat: Agents are commoditizing; persistent, searchable knowledge is rare
    • Identity drift risk: "Agent workspace" makes EMDX sound like "yet another AI dev tool"
    • Alternative framing: "EMDX is a knowledge base that agents can read and write" keeps KB central

    Patterns from Other Tools¶

    The dual-interface architecture draws from proven CLI tools:

    Git's Porcelain Pattern:

    git status              # Human: "On branch main, nothing to commit"
git status --porcelain  # Machine: "?? newfile.txt" (stable format)

    Ripgrep's Structured Output:

    rg "pattern" --json     # Each match as JSON object (JSON Lines format)
rg "pattern" --quiet    # Exit code only, no output

    gh CLI's Template-Based Formatting:

    gh pr list --json number,title,author
gh pr list --json number,title --jq '.[].number'

    Agent Detection Implementation¶

    def is_agent_context() -> bool:
    """Detect if running in an AI agent context."""
    if os.environ.get("CLAUDECODE") == "1":
        return True
    if os.environ.get("EMDX_AGENT_MODE") == "1":
        return True
    if not sys.stdin.isatty() and not sys.stdout.isatty():
        return True
    return False

    Structured Error Output¶

    When --json is specified, errors output as structured data:

    {
  "success": false,
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "Document with ID 999 not found",
    "details": {"requested_id": 999}
  }
}

    Error codes follow HTTP-like semantics: - INVALID_INPUT, DOCUMENT_NOT_FOUND, PERMISSION_DENIED (4xx equivalent) - DATABASE_ERROR, NETWORK_ERROR, UNKNOWN_ERROR (5xx equivalent)

    Implementation Details¶

    OutputMode Enum¶

    from enum import Enum

class OutputMode(Enum):
    HUMAN = "human"      # Rich tables, colors, emojis
    JSON = "json"        # Full structured JSON
    QUIET = "quiet"      # Just essential values (IDs)
    PORCELAIN = "porcelain"  # Stable text format, no breaking changes

def detect_output_mode(
    json_flag: bool = False,
    quiet_flag: bool = False,
    porcelain_flag: bool = False,
    auto_detect: bool = True
) -> OutputMode:
    """Determine output mode from flags and environment."""
    if json_flag:
        return OutputMode.JSON
    if quiet_flag:
        return OutputMode.QUIET
    if porcelain_flag:
        return OutputMode.PORCELAIN

    if auto_detect and not sys.stdout.isatty():
        return OutputMode.QUIET

    return OutputMode.HUMAN

    Standardized Exit Codes¶

    Code Meaning
    0 Success
    1 General error
    2 Invalid arguments/input
    3 Resource not found
    4 Permission denied
    5 Network/external service error

    Environment Variables¶

    Variable Purpose
    CLAUDECODE=1 Claude Code is running (auto-set)
    EMDX_AGENT_MODE=1 Generic agent mode marker
    EMDX_OUTPUT_MODE=json\|quiet\|porcelain Override default output mode
    EMDX_NO_COLOR=1 Disable color output
    EMDX_NO_PAGER=1 Disable pager

    What Changes for Agents (Default Behavior)¶

    Behavior Human Default Agent Default
    Output mode Rich tables Quiet (IDs only)
    Confirmations Prompt user Skip (assume yes)
    Pager Use for long output No pager
    Progress bars Show spinners Hide or minimal
    Color Auto (term-aware) Off

    Critical principle: These are defaults, not mandates. Explicit flags always override.

    Agent Workflow Integration¶

    Session Start Protocol¶

    emdx prime    # Get current work context (ready tasks, in-progress, recent docs)
emdx status   # Quick overview

    With hooks active in Claude Code, emdx prime runs automatically on session start.

    Briefing System¶

    emdx briefing                        # What happened in last 24h
emdx briefing --since "2 days ago"
emdx briefing --save                 # AI summary, persisted to KB

    This addresses the critical gap: humans need to consume what agents produced across sessions.

    Review Queue Workflow¶

    Future implementation to support agenthumanagent handoffs:

    emdx review list                     # Show items awaiting human decision
emdx review approve 42
emdx review reject 42 --reason "focus on security first"

    Continue-with-Feedback Pattern¶

    Planned feature for iterative agent work:

    emdx delegate --continue-from 42 --feedback "prioritize performance over readability"

    Missing Features Analysis¶

    For Humans (consuming agent output)¶

    • Briefing system: Synthesize what happened across sessions (emdx briefing --since)
    • Activity feeds: Semantic grouping by session/topic, not flat chronological
    • Quality signals: No way to mark "verified by human" vs "agent draft"
    • Diff/delta views: Document evolution tracking

    For Agents (efficient operation)¶

    • Semantic context injection: --context "topic" auto-selects relevant docs
    • Structured schemas: Validation for agent outputs
    • Relationship tracking: Explicit links between documents
    • Agent memory: Learning from similar past tasks

    For Handoffs (the critical gap)¶

    • Review queue: First-class pendingapproved/rejected workflow
    • Structured review requests: Agent asks specific questions
    • Continue pattern: Resume with human feedback incorporated
    • Notification system: Escalation and priority handling

    The biggest gap: EMDX treats documents as isolated artifacts, but the workflow connecting them is implicit.

    Implementation Roadmap¶

    Phase 1: Foundation (Immediate)¶

    1. Create OutputMode enum and detect_output_mode() utility
    2. Add --json, -q, --porcelain as global CLI options
    3. Standardize structured error output with error codes
    4. Migrate high-traffic commands: save, find, view, tag, prime

    Phase 2: Consistency (Near-term)¶

    1. Structured errors everywhere
    2. Standardize all list/search commands for JSON output
    3. Document porcelain formats as versioned API
    4. Implement emdx briefing --since for human consumption
    5. Add review queue workflow

    Phase 3: Advanced (Future)¶

    1. JSON input support (--from-json for complex operations)
    2. Semantic context injection (--context flag)
    3. Structured output schemas with validation
    4. Continue-with-feedback pattern
    5. Agent memory/learning system

    Design Principles¶

    The dual-interface architecture follows these core principles:

    1. Explicit beats implicit Flags override auto-detection
    2. Humans are the default Don't optimize for machines at human expense
    3. Stability promises Porcelain formats are versioned API contracts
    4. Fail loudly Structured errors with codes, not silent failures
    5. Pipe-friendly Every command that outputs data should be pipeable
    6. Composable Commands work together via stdin/stdout
    7. Workflow, not just artifacts Support the full cycle, not just storage

    Related Topics¶

    Competitive Landscape (2025-2026)¶

    MCP as Universal Standard: Model Context Protocol has become industry standard, with Google, Microsoft, AWS, and OpenAI all supporting it. By 2026, 75% of API gateway vendors will have MCP features.

    Memory is Table Stakes: Every major coding agent now has persistent state: - Claude Code: Task system with DAGs, resumable subagents, memory files - Cursor: Persistent knowledge base, automatically generated memories - Windsurf: User-generated and automatically generated memory systems - Devin: Automatic repository indexing with architecture diagrams

    Agent Workspaces Emerging: Google Workspace Studio, Amazon Bedrock AgentCore, Microsoft agent workspaces all provide dedicated environments for agent operation.

    Graph-Based Orchestration: LangGraph's checkpoint model becoming production standard, with PostgreSQL persistence used at LinkedIn, Uber, and 400+ companies.

    Why Hierarchy Serves Both Humans and Agents¶

    For Humans: - Cognitive scaffolding for limited working memory (Miller's 72 items) - Navigation by recognition vs recall - Context maintenance ("I'm working on SEC stuff") - Scope communication ("The auth epic is 80% done")

    For Agents: - Semantic compression for context windows - Scoped queries for efficiency - Delegated authority patterns ("handle all SEC tasks autonomously")

    The Positioning Statement¶

    "EMDX is the knowledge layer for AI-assisted developmenta persistent, searchable, human-controllable system that any agent can read from and write to."

    This positions EMDX as infrastructure, not just another tool. It's the memory that makes agent work valuable beyond single sessions.
