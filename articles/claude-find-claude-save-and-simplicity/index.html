    claude-find, claude-save, and simplicity¶

    Overview¶

    The clauding project evolved through multiple architectural iterations, repeatedly learning the same lesson: simplicity beats features. This wiki documents the core commands (claude-save, claude-find), the journey from complexity to clarity, and the Fish plugin architecture that finally made the system "just work."

    Key Concepts¶

    The Core Workflow¶

    Clauding provides three essential operations:

    1. Save: claude-save <file> or claude-clip (from clipboard)
    2. Find: claude-find <search> or claude-recent
    3. View: claude-view <id> with tab completion

    This mirrors the simplest knowledge capture workflow: save something, search for it later, read it when needed. The system is designed specifically for Claude Code workflows where building context through markdown files is essential.

    Search-First Architecture¶

    PostgreSQL full-text search (FTS5) powers the find operation with: - GIN indexes for instant search - Weighted ranking (title vs content) - Auto-updating search vectors via triggers - Millisecond response times for thousands of documents

    The search vector updates automatically on every insert/update, requiring no manual index maintenance.

    Architecture & Design Decisions¶

    The Plugin Architecture Journey¶

    Clauding underwent a major refactor from a manual-sourcing system to a standard Fish plugin. The transformation addressed critical pain points:

    Before (Manual Sourcing): - Required sourcing 3+ files in exact order - claude-find defined in 3 different files (standard, enhanced, scale) - Files named *-fixed.fish indicated hasty patches - Documentation referenced non-existent files like core/claude-init.fish - Broken installation scripts creating relative symlinks

    The Simplicity Principle¶

    Multiple design documents championed stripping away complexity:

    Version Tracking evolved from: - Complex metadata tables, content hashing, deduplication strategies, compression - To: Simple trigger storing old content before updates, full snapshots each time

    Document Relationships were planned but deferred until the core workflow proved insufficient.

    Categories and Tags were minimized in favor of full-text search across all content.

    The MVP document crystallized this philosophy:

    "No complex tagging, no projects, no threading. Just save, find, view... You might not need anything else."

    Why Claude Code Agents Were Struggling¶

    When the system moved to plugin mode without updating documentation, agents failed because: 1. CLAUDE.md referenced non-existent files (core/claude-init.fish) 2. dev-install.fish used relative paths: set -l script_dir (dirname (status -f)) returned "." instead of absolute path 3. Broken symlinks appeared to install successfully but functions never loaded 4. Internal functions in _clauding_internal/ weren't in Fish's function path

    The fix required three changes:

    # Fix in dev-install.fish
set -l script_dir (realpath (dirname (status -f)))

# Fix in conf.d/clauding.fish  
set -l clauding_root (dirname (dirname (status -f)))
if test -d $clauding_root/functions/_clauding_internal
    set -p fish_function_path $clauding_root/functions/_clauding_internal
end

    Implementation Details¶

    Minimal Schema¶

    CREATE SCHEMA claude;

CREATE TABLE claude.knowledge (
    id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    category TEXT DEFAULT 'general',
    search_vector tsvector
);

CREATE INDEX idx_claude_created ON claude.knowledge(created_at DESC);
CREATE INDEX idx_claude_search ON claude.knowledge USING GIN(search_vector);

CREATE TRIGGER update_search_vector 
BEFORE INSERT OR UPDATE ON claude.knowledge
FOR EACH ROW EXECUTE FUNCTION
tsvector_update_trigger(search_vector, 'pg_catalog.english', title, content);

    Function Consolidation¶

    The merged claude-find supports optional flags instead of maintaining separate implementations:

    claude-find "search term"                    # Basic search
claude-find --snippets "search term"         # With context
claude-find --fuzzy "search term"            # Fuzzy matching
claude-find --project api --limit 5 "term"   # Filtered search

    Auto-Loading via conf.d¶

    The conf.d/clauding.fish file loads automatically when Fish starts:

    # Set up database connection
set -g CLAUDING_DB "knowledge_base"
set -g CLAUDING_USER "$USER"

# Define internal database function
function _clauding_db
    psql -U $CLAUDING_USER -d $CLAUDING_DB -t -A -c $argv
end

# Add internal functions to path
set -l clauding_root (dirname (dirname (status -f)))
if test -d $clauding_root/functions/_clauding_internal
    set -p fish_function_path $clauding_root/functions/_clauding_internal
end

    This eliminates all manual sourcing requirements.

    Related Topics¶

    The "Personal OS" Split¶

    An early vision treated clauding as a "Personal OS" including email extraction, financial documents, and private document management. The recommendation was to split into:

    1. Claude-KB - Universal knowledge capture for Claude workflows (this project)
    2. Personal OS - Private email/financial systems

    The key insight: claude-save solved a specific problem (building context for Claude Code sessions) and diluting it with unrelated features would obscure that value.

    Distribution Strategy¶

    Proposed paths to adoption: - Switch from PostgreSQL to SQLite for zero-config distribution - Single-file Fish function for easy sourcing - Multi-shell support (bash, zsh, Python CLI) - Fisher plugin: fisher install arockwell/claude-kb

    The magic formula: Simple Install + Immediate Value + Claude Integration = Adoption

    Command Structure Patterns¶

    Analysis of command signatures revealed consistent patterns:

    Input sources: - File-based: claude-save <file> - Direct text: claude-direct <title> <content> - Piped: echo "text" | claude-pipe <title> - Clipboard: claude-clip [title] - Command output: claude-cmd <command>

    Parameter patterns: - Positional arguments for primary inputs - Optional arguments with smart defaults - Auto-detection when parameters omitted (e.g., title from filename, project from git)

    This consistency makes the system intuitive: all commands follow claude-<verb> naming, and similar operations accept similar parameters.
