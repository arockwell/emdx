    browse.py, trash.py, and tags.py Command Layer Architecture¶

    Overview¶

    The browse.py, trash.py, and tags.py modules comprise EMDX's document management and organization CLI layer. Together they provide ~880 lines of user-facing command implementations for browsing documents, managing soft-deleted content, and organizing content with tags. All three follow the Typer framework pattern with Rich library formatting for human-readable output and JSON modes for programmatic access.

    browse.py: Document Browser and Statistics¶

    Purpose: Interactive TUI browser entry point and document analytics.

    Core Commands: - emdx browse / emdx gui Launch Textual TUI for interactive document/task browsing - emdx browse list List all documents with filtering by project - emdx browse recent Show most recently accessed documents - emdx browse stats Display knowledge base statistics with project breakdowns - emdx view <id> Display full document content with metadata - emdx edit <id> Open document in $EDITOR - emdx delete <id> Move document to trash (soft delete)

    TUI Integration¶

    CRITICAL WARNING: Never run emdx gui in Claude Code sessions the TUI launches an interactive full-screen application that captures terminal input, causing sessions to hang. This is documented in CLAUDE.md.

    The TUI mode calls run_browser() from emdx/ui/run_browser.py, which initializes Textual's BrowserApp with multiple browser modes: - Document browser (default) Navigate document hierarchy - File browser Watch and save files - Git browser Repository activity - Log browser Execution logs - Task browser Task management - Agent browser Agent configurations

    Statistics and Analytics¶

    Document statistics are grouped by project and stage:

    SELECT project, stage, COUNT(*) 
FROM documents 
GROUP BY project, stage

    When --detailed flag is used, shows access pattern histograms:

    CASE
    WHEN access_count = 0 THEN 'Never accessed'
    WHEN access_count <= 5 THEN '1-5 views'
    WHEN access_count <= 10 THEN '6-10 views'
    ...
END as view_range

    Output Modes¶

    Following the CLI Output Mode Convention: - Default: Plain text output (machine/pipe friendly) - --rich flag: Colored Rich markup (panels, syntax highlighting, markdown rendering) - --json flag: Structured machine output - Uses print() for plain output, console.print() only in --rich mode

    Refactoring (PR #565)¶

    Adopted @combined_decorator and require_document() helpers, removing ~30 lines of boilerplate:

    Before:

    try:
    db.ensure_schema()
    doc = get_document(identifier)
    if not doc:
        console.print("[red]Error: Document not found[/red]")
        raise typer.Exit(1)
    # ... command logic
except Exception as e:
    console.print(f"[red]Error: {e}[/red]")
    raise typer.Exit(1)

    After:

    @combined_decorator("viewing document")
def view(identifier: str):
    doc = require_document(identifier)
    # ... command logic

    trash.py: Soft Delete Lifecycle Management¶

    Purpose: Trash/restore/purge operations with batch support and time-based filtering.

    Core Commands: - emdx trash <ids...> Soft delete documents (sets deleted_at timestamp) - emdx trash list Show trashed documents - emdx restore <ids...> Restore documents from trash - emdx trash empty Permanently delete trashed documents

    Batch Operations¶

    All commands support flexible ID specification: - Multiple IDs: emdx trash 42 43 44 - ID ranges: emdx trash 10-20 (expands to [10, 11, ..., 20]) - Mixed syntax: emdx trash 5 10-12 20 [5, 10, 11, 12, 20] - --all flag: emdx trash restore --all (applies to all trashed documents)

    Time-Based Filtering¶

    Optional aging filters:

    # List only recently trashed
days: int | None = typer.Option(None, "--days", "-d")

# Purge only old items
older_than: int | None = typer.Option(None, "--older-than")

    Example: emdx trash empty --older-than 30 removes documents trashed more than 30 days ago.

    Safety Mechanisms¶

    Soft Delete Pattern: - Sets deleted_at timestamp (preserves data) - Clears deleted_at on restore - Documents remain searchable in trash views

    Purge Warnings: Before permanent deletion (empty command): - Shows document count to be deleted - Uses Confirm.ask() from Rich for confirmation - Requires --force flag to skip prompt - Warning text: "This action cannot be undone!"

    Database Operations:

    UPDATE documents SET deleted_at = CURRENT_TIMESTAMP WHERE id = ?  # Trash
UPDATE documents SET deleted_at = NULL WHERE id = ?               # Restore
DELETE FROM documents WHERE id = ?                                # Purge

    Purge performs cascading cleanup: - Removes document row - Deletes associated tags (DELETE FROM document_tags WHERE document_id = ?) - Removes embeddings if index exists - Cleans up linked files

    Refactoring (PR #565)¶

    Removed ~40 lines of boilerplate by adopting command helpers. Commands are now <10 lines each:

    @trash_app.command("restore")
@combined_decorator("restoring document")
def restore_command(identifier: str) -> None:
    doc = require_document(identifier)
    restore_document(doc["id"])
    console.print(f"[green] Restored document #{doc['id']}: {doc['title']}[/green]")

    tags.py: Flexible Tag Management System¶

    Purpose: Tag CRUD operations with emoji aliases, ML-powered suggestions, and batch auto-tagging.

    Core Commands: - emdx tag add <id> [tags...] Manually add tags to a document - emdx tag add <id> --auto Apply high-confidence auto-suggested tags - emdx tag add <id> --suggest Show ML-based tag suggestions with confidence scores - emdx tag remove <id> [tags...] Remove tags from a document - emdx tag list Show all tags with usage counts - emdx tag legend Display emoji-to-text-alias mapping - emdx tag rename <old> <new> Rename tag globally across all documents - emdx tag merge [tags...] --into <target> Merge multiple tags into one - emdx tag batch Batch auto-tag untagged documents with ML suggestions

    Emoji Alias System¶

    Text-to-emoji conversion via emdx/config/tagging_rules.py:

    TAG_CONFIG = {
    "gameplan": "",
    "active": "",
    "done": "",
    "blocked": "",
    "success": "",
    "failed": "",
    "analysis": "",
    "security": ""
}

    Users can specify tags as plain text or emojis; normalize_tag() converts both to canonical emoji form for storage:

    emdx tag add 42 gameplan active    # Text aliases
emdx tag add 42                # Direct emoji input
# Both result in same stored tags: ["", ""]

    Auto-Tagging Integration¶

    Uses AutoTagger service for ML-powered suggestions:

    from emdx.services.auto_tagger import AutoTagger

tagger = AutoTagger()
suggestions = tagger.suggest_tags(doc_id)
# Returns: [(tag_name, confidence), ...]

    Batch Auto-Tag Workflow: 1. Analyze all untagged documents (or filtered subset) 2. Generate tag suggestions with confidence scores 3. Filter by confidence threshold (default 0.7 = 70%) 4. Show preview of first 10 documents 5. Dry-run mode (default) or execute with --execute 6. Progress spinner during application

    Example:

    $ emdx tag batch --confidence 0.8 --execute
Analyzing 45 untagged documents...
Found 32 documents with high-confidence suggestions
Applying tags to 32 documents...  100%
 Applied 87 tags across 32 documents

    Database Schema¶

    Many-to-many relationship:

    CREATE TABLE tags (
    id INTEGER PRIMARY KEY,
    name TEXT UNIQUE NOT NULL
);

CREATE TABLE document_tags (
    document_id INTEGER,
    tag_id INTEGER,
    FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
    FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE,
    UNIQUE(document_id, tag_id)
);

    Tag counting query pattern:

    SELECT t.name, COUNT(*) as count 
FROM tags t
JOIN document_tags dt ON t.id = dt.tag_id
GROUP BY t.name 
ORDER BY count DESC

    Multi-Tag Search¶

    From CLI Reference: - --tags "gameplan,active" Comma-separated = AND logic (both tags required) - --any-tags flag Enables OR logic (any tag matches)

    Important limitation: - emdx find does NOT support OR/AND/NOT operators in query string - escape_fts5_query() quotes each term, making operators literal - Use separate emdx find calls or --tags with --any-tags for OR logic

    Tags vs Categories (CRITICAL DISTINCTION)¶

    Documents use plain text tags for flexible classification: - Tags: gameplan, active, analysis, security, blocked - Status tracking: active (working), done (completed), blocked (stuck) - Outcome: success, failed, partial

    Tasks use categories and epics (NOT tags): - --cat FEAT / --cat FIX / --cat ARCH for categorization - --epic <id> for grouping under parent epic - emdx task cat list to see available categories - emdx task epic list to see active epics

    This separation allows flexible document organization while maintaining structured project management.

    Refactoring (PR #565)¶

    Removed ~60 lines of error handling boilerplate across 6 commands by adopting @combined_decorator + require_document().

    Architecture and Design Patterns¶

    1. Command Layer Responsibilities¶

    All three modules follow standard CLI layering: - Input validation Parse Typer arguments, validate ID syntax, check flags - Business logic delegation Call database/model/service functions - Output formatting Render Rich tables/panels or JSON

    None execute raw SQL they delegate to typed functions in emdx/database/: - browse.py get_documents(project=project), get_recent_documents() - trash.py soft_delete_document(), restore_document(), hard_delete_documents() - tags.py add_tags(), remove_tags(), get_all_tags(project=project)

    2. Shared Infrastructure¶

    All three modules leverage: - Shared console: from emdx.utils.output import console (Track B deduplication) - Error handling decorator: @handle_cli_errors from emdx.utils.cli - Command helpers: @combined_decorator, require_document() from emdx.commands._helpers - Database transactions: All multi-step operations wrapped with rollback safety - Rich/Plain output modes: --rich flag for human-readable formatting

    3. Hybrid Identifier Resolution¶

    All three support flexible document identification:

    def get_document(identifier: str | int) -> dict | None:
    """Fetch by ID (int) or title substring (str)."""
    if isinstance(identifier, int):
        return _fetch_by_id(identifier)
    return _search_by_title_substring(identifier)

    Examples:

    emdx view 42                    # By ID
emdx view "architecture notes"  # By title substring
emdx trash restore 42-50        # ID range

    4. Text Truncation¶

    All use canonical truncate() function from emdx/utils/text_formatting.py:

    from emdx.config.constants import TITLE_TRUNCATE_LENGTH
from emdx.utils.text_formatting import truncate

title = truncate(doc['title'], TITLE_TRUNCATE_LENGTH)  # Max 50 chars

    Constants: - TITLE_TRUNCATE_LENGTH = 50 - DESCRIPTION_TRUNCATE_LENGTH = 40 - PATH_TRUNCATE_LENGTH = 35

    5. Optional Filters as | None¶

    All filter parameters use | None type annotations with typer.Option(None, ...): - browse.py: project: str | None - trash.py: days: int | None, older_than: int | None - tags.py: project: str | None, limit: int | None

    This pattern is necessary and intentional None represents "no filter applied". The database layer interprets None as "include all" and builds SQL queries accordingly. From the None Type Analysis audit: 0 bugs found all | None annotations match runtime behavior.

    Code Quality and Technical Debt¶

    Strengths¶

    • Clear separation of concerns: Commands handle I/O, database handles persistence
    • Rich user feedback: Emojis, colors, progress spinners enhance UX
    • Safety-first design: Destructive operations require confirmation
    • Multi-format output: Supports human (Rich) and machine (JSON/CSV) readers
    • Comprehensive tests: All three have test coverage in tests/test_commands_*.py

    Technical Debt (from Architecture Analysis #6330)¶

    RESOLVED (PR #565): - Command helpers adopted @combined_decorator eliminates ~130 lines of boilerplate - Document validation centralized require_document() replaces repeated patterns - Error handling standardized consistent error messages across all commands

    REMAINING: 1. Type safety: Still use raw dicts from sqlite3.Row no TypedDict definitions 2. Service layer bypass: browse.py uses db.get_connection() directly for stats (should go through service) 3. Duplicate datetime parsing: Three separate implementations across database modules 4. Global state dependency: All rely on db_connection singleton, complicating test isolation

    Recommended Improvements¶

    1. Define TypedDict for Document:

    from typing import TypedDict

class DocumentDict(TypedDict):
    id: int
    title: str
    content: str
    project: str | None
    created_at: datetime
    deleted_at: datetime | None

    2. Centralize datetime parsing:

    # utils/datetime_utils.py
def parse_sqlite_datetime(s: str | None) -> datetime | None:
    if not s:
        return None
    return datetime.fromisoformat(s)

    3. Consistent JSON output flag: Current state: - browse.py: --format json - tags.py: --json - trash.py: No JSON mode

    Should standardize on --json across all three.

    Integration Points¶

    With Database Layer¶

    • emdx.database.core CRUD operations
    • emdx.database.search Full-text search via FTS5
    • emdx.database.links Link graph queries
    • emdx.database.documents Document lifecycle (soft delete, restore, access tracking)

    With Services Layer¶

    • tags.py services/auto_tagger.py for ML-based tag suggestions
    • browse.py services/document_manager.py for health scoring and batch operations

    With UI Layer¶

    • browse.py ui/run_browser.py BrowserApp (Textual TUI)
    • browse.py ui/document_browser.py for document list view (1122 LOC)

    With Utils¶

    • utils/text_formatting.truncate() Text truncation with ellipsis
    • utils/tags.normalize_tag() Emoji alias resolution
    • utils/cli.handle_cli_errors Standardized error display

    Performance Characteristics¶

    • browse.py JSON mode: O(n) scan when no filter applied can be slow on large KBs (10k+ docs). Use --project to scope.
    • trash.py batch operations: Uses executemany() for soft deletes (fast), but hard deletes are sequential (~100ms per document for large purges)
    • tags.py list: JOIN query is indexed on document_tags.tag_id, fast even with 100k+ tag assignments. Defaults to limit of 50 for readable output.

    Testing Coverage¶

    From test coverage audit (#6162): - browse.py tests/test_browse.py, tests/test_commands_core.py - tags.py tests/test_tags.py, tests/test_commands_tags.py - trash.py NO test coverage (identified gap)

    Recommendation: Priority 2 for trash.py testing (user-facing document lifecycle command).
