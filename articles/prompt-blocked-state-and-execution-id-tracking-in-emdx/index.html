    Prompt, Blocked State, and Execution ID Tracking in EMDX¶

    Overview¶

    EMDX manages AI agent execution through a three-layer architecture that separates work semantics (tasks), process tracking (executions), and their relationships (task_executions). This system enables dependency-aware task scheduling, comprehensive execution tracking, and persistent prompt storage for reproducibility.

    Core Architecture¶

    Three-Layer Model¶

    Tasks Table â€” Semantic work units with full context: - Stores prompts, dependencies, hierarchies, and outcomes - Tracks human-readable status: open, active, blocked, done, failed - Links to source documents (source_doc_id) and output documents (output_doc_id) - Supports hierarchies via parent_task_id and ordering via seq

    Executions Table â€” Process-level tracking: - Records every Claude API invocation with PID, log file, resource usage - Tracks runtime status: running, completed, failed - Stores cost analytics: cost_usd, input_tokens, output_tokens - Can exist independently of tasks (e.g., GUI sessions)

    Task Executions Table â€” Join layer: - Links tasks to executions, supporting multiple attempts per task - Tracks execution type: direct (Claude run), manual (human completion) - Originally designed for retry tracking but bypassed by current delegate implementation

    Prompt Storage and Flow¶

    Where Prompts Live¶

    Database Storage:

    # tasks.prompt field (TEXT, nullable)
# Truncated to 500 chars during task creation
task_id = create_task(
    prompt=prompt[:500],  # For UI display, search
    ...
)

    Execution Use:

    # Full prompt passed to UnifiedExecutor
config = ExecutionConfig(
    prompt=prompt,  # Untruncated, actually sent to API
    ...
)

    Why Two Copies: - tasks.prompt (truncated): Database efficiency, UI display, task lists - Execution prompt (full): Actually sent to Claude, not permanently stored - Full prompt lost after execution unless captured in log file

    Prompt Sources¶

    Single Task:

    emdx delegate "analyze the auth module"
# tasks.prompt = "analyze the auth module"

    With Document Context:

    emdx delegate --doc 42 "implement the plan"
# Prompt includes document content from doc #42

    Chain Tasks: - Each step receives previous step's output via source_doc_id - Output document of step N becomes source document of step N+1 - Enables context accumulation across pipeline

    Session Context: - emdx prime (run by .claude/hooks/prime.sh) injects ready tasks, in-progress work - Sets EMDX_TASK_ID, EMDX_DOC_ID environment variables - Claude receives context automatically on session start

    When Tasks Become Blocked¶

    Dependency Mechanism:

    Tasks block when they have unsatisfied dependencies in the task_deps table:

    CREATE TABLE task_deps (
    task_id INTEGER REFERENCES tasks(id) ON DELETE CASCADE,
    depends_on INTEGER REFERENCES tasks(id) ON DELETE CASCADE,
    PRIMARY KEY (task_id, depends_on)
);

    Ready Task Query (from get_ready_tasks() in models/tasks.py):

    SELECT t.* FROM tasks t
WHERE t.status = 'open'
AND NOT EXISTS (
    SELECT 1 FROM task_deps d
    JOIN tasks dep ON d.depends_on = dep.id
    WHERE d.task_id = t.id AND dep.status != 'done'
)
ORDER BY priority, created_at

    Key Insight: The blocked status is explicitly set, not automatically derived. Tasks with unsatisfied dependencies simply won't appear in get_ready_tasks() even if their status is open.

    Blocking Scenarios: 1. Chain dependencies: Step N+1 depends on step N completing 2. Manual dependencies: Added via emdx task depends <id> --on <prereq_id> 3. External blockers: Manual emdx task update <id> -s blocked for human input 4. Failed prerequisites: Dependents must be manually updated

    Cycle Detection¶

    Before adding a dependency, EMDX runs depth-first search to prevent cycles:

    def _would_cycle(task_id: int, new_dep: int) -> bool:
    """Check if adding dependency would create cycle."""
    visited, stack = set(), [new_dep]
    while stack:
        current = stack.pop()
        if current == task_id:  # Found path back to original task
            return True
        if current in visited:
            continue
        visited.add(current)
        # Follow all dependencies of current task
        stack.extend(get_dependencies(current))
    return False

    Execution ID Lifecycle¶

    Execution Record Creation¶

    Schema:

    CREATE TABLE executions (
    id INTEGER PRIMARY KEY,
    doc_id INTEGER,              -- Often NULL for delegates!
    doc_title TEXT,              -- "Delegate: {prompt}" or custom
    status TEXT,                 -- running/completed/failed
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    log_file TEXT,
    exit_code INTEGER,
    pid INTEGER,
    cost_usd REAL,
    input_tokens INTEGER,
    output_tokens INTEGER,
    ...
);

    Created When:

    # unified_executor.py creates execution before task starts
exec_id = create_execution(
    doc_id=None,  # No output doc yet!
    doc_title=f"Delegate: {prompt[:60]}",
    status="running",
    log_file=log_path,
    pid=process.pid
)

    Linking Task to Execution¶

    Delegate Execution Flow:

    1. Task Created (before execution): python task_id = create_task( prompt=prompt[:500], status="active", execution_id=None # Not linked yet )

    2. Execution Starts: python result = UnifiedExecutor().execute(config) # Creates execution record with running status

    3. Task Linked to Execution: python update_task(task_id, execution_id=result.execution_id)

    4. Agent Saves Output:

    5. Claude runs emdx save inside session

    6. Returns document ID via stdout

    7. Task Completed: python update_task( task_id, status="done", output_doc_id=doc_id, completed_at=datetime.now(timezone.utc) )

    8. Execution Completed (automatic): python update_execution(exec_id, status="completed", exit_code=0)

    Forward and Reverse Links¶

    Forward Link â€” tasks.execution_id â†’ executions.id: - Set immediately after execution starts - Enables task-to-log navigation - Query pattern: sql SELECT t.*, e.log_file, e.cost_usd FROM tasks t JOIN executions e ON t.execution_id = e.id WHERE t.id = ?

    Reverse Link â€” executions.id â† tasks.execution_id: - No FK from executions to tasks - Query uses join on task's FK: sql SELECT e.*, t.output_doc_id, t.tags FROM executions e LEFT JOIN tasks t ON t.execution_id = e.id WHERE e.started_at > ?

    Multiple Tasks per Execution: Theoretically possible (multiple tasks could reference same execution_id), though delegate doesn't use this pattern.

    Hierarchical Task Execution¶

    Single Tasks¶

    emdx delegate "analyze auth module"

    Creates: - 1 task (type='single', status='active') - 1 execution (log file, PID, cost tracking) - 1 output document (when agent saves)

    Group Tasks (Parallel)¶

    emdx delegate "task1" "task2" "task3" -j 3

    Creates: - 1 parent task (type='group', status='active') - 3 child tasks (parent_task_id set, seq=0,1,2, type='single') - 3 separate executions (run in parallel via ThreadPoolExecutor)

    Progress Tracking:

    SELECT t.*,
    (SELECT COUNT(*) FROM tasks c WHERE c.parent_task_id = t.id) as child_count,
    (SELECT COUNT(*) FROM tasks c WHERE c.parent_task_id = t.id AND c.status = 'done') as children_done
FROM tasks t
WHERE t.status = 'active' AND t.parent_task_id IS NULL

    Chain Tasks (Sequential)¶

    emdx delegate --chain "analyze" "plan" "implement"

    Creates: - 1 parent task (type='chain') - 3 child tasks with dependencies: - Step 1: no dependencies - Step 2: depends on step 1 - Step 3: depends on step 2

    Dependency Wiring:

    prev_task_id = None
for i, prompt in enumerate(prompts):
    task_id = create_task(
        prompt=prompt,
        parent_task_id=parent_id,
        seq=i,
        source_doc_id=prev_output_doc_id  # Chain output
    )
    if prev_task_id:
        add_dependency(task_id, prev_task_id)
    prev_task_id = task_id

    Output Chaining: - Step N's output_doc_id becomes Step N+1's source_doc_id - Document content injected into next step's prompt - Pipeline stops if any step fails

    Data Inconsistencies¶

    Issue 1: doc_id Usually NULL in Executions¶

    Problem: Executions created before output documents exist:

    UnifiedExecutor.execute() â†’ create_execution(doc_id=NULL)
  â†’ Agent runs, saves document
  â†’ Delegate updates task.output_doc_id
  â†’ execution.doc_id stays NULL!

    Workaround (from execution_service.py):

    SELECT e.id, e.doc_id, t.output_doc_id
FROM executions e
LEFT JOIN tasks t ON t.execution_id = e.id

    Then coalesce: doc_id = e.doc_id or t.output_doc_id

    Issue 2: Prompt Truncation¶

    Problem: tasks.prompt only stores first 500 characters:

    prompt=prompt[:500],  # Information loss!

    Impact: Long prompts cannot be fully reconstructed from task records. Full prompt is lost after execution unless stored in log file.

    Issue 3: task_executions Bypassed¶

    Finding: Delegate creates tasks with execution_id but never populates task_executions table.

    Evidence: - delegate.py calls create_task() with execution_id - Never calls create_task_execution() - Table still references removed workflow_run_id

    Conclusion: Legacy infrastructure from removed workflow system. Current delegate bypasses it entirely.

    Issue 4: Custom Titles Break Activity Browser¶

    Problem: Activity browser filters by title pattern:

    WHERE (e.doc_title LIKE 'Agent:%' OR e.doc_title LIKE 'Delegate:%')

    Custom titles via --title won't match and won't appear in Activity view.

    Retry Mechanism¶

    Retry Schema¶

    tasks.retry_of INTEGER REFERENCES tasks(id)

    Retry Flow¶

    1. Task Fails: python update_task(task_id, status='failed', error="error message")

    2. User Retries: bash emdx task retry 42

    3. New Task Created: python new_task_id = create_task( title=f"Retry: {original.title}", prompt=original.prompt, retry_of=original.id, source_doc_id=original.source_doc_id, gameplan_id=original.gameplan_id )

    4. New Execution:

    5. Fresh execution_id assigned
    6. Original task preserves failure history
    7. Retry chain tracked via retry_of FK

    Key Insight: Retries create NEW tasks, not updates. This preserves full execution history.

    Execution vs Task Status¶

    Status Divergence¶

    Tasks track semantic state: - open: Ready to work - active: Being worked on - blocked: Waiting on dependencies/external factors - done: Successfully completed - failed: Execution error

    Executions track runtime state: - running: Process active - completed: Process exited successfully - failed: Process exited with error

    Divergence Example:

    # Task starts
task = create_task(status="active")
execution = create_execution(status="running")

# External API goes down mid-execution
update_task(task_id, status="blocked")
# execution.status remains "running" until process exits

    Blocked vs Failed¶

    State Meaning Recovery
    blocked Temporarily stuck, may resume Wait for dependencies or manual intervention
    failed Execution terminated with error Requires retry with --retry or fixes
    open with deps Implicitly blocked by dependencies Automatic when deps complete

    Recommendations¶

    1. Store Full Prompts¶

    Options: - Remove 500-char truncation (disk is cheap) - Add executions.prompt column - Store full prompt in log file header

    2. Populate execution.doc_id After Completion¶

    # When delegate gets output doc_id
update_task(task_id, output_doc_id=doc_id)
update_execution(execution_id, doc_id=doc_id)  # Add this!

    Eliminates join workaround in activity browser.

    3. Consolidate or Remove task_executions¶

    Option A: Remove entirelyâ€”delegate doesn't use it Option B: Use for retry trackingâ€”each attempt creates record

    4. Fix Activity Browser Filtering¶

    Replace title pattern matching with explicit tracking:

    ALTER TABLE executions ADD COLUMN source_type TEXT;

    Summary¶

    The EMDX prompt and execution tracking system provides:

    Prompt Storage: - Tasks store prompts (truncated to 500 chars) for UI/search - Full prompts sent to Claude during execution - Context assembled from tasks, documents, gameplans, chain outputs

    Execution Tracking: - Three-layer architecture: tasks (semantics), executions (processes), task_executions (joins) - execution_id links tasks to runtime logs, costs, tokens - Separate status tracking: tasks (work state), executions (runtime state)

    Dependency Management: - DAG-based dependencies via task_deps table - Cycle detection prevents invalid dependency graphs - Ready task query excludes blocked tasks automatically - Chain tasks create linear pipelines with automatic dependencies

    Key Pattern: Tasks bridge semantic work tracking (prompts, dependencies, blocking) with runtime execution tracking (logs, PIDs, exit codes). The execution_id field links these two layers, enabling EMDX to track "what to do" separately from "how it's being done."
