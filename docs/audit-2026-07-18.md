# Codebase Audit — 2026-07-18

Full parallel audit of `emdx/` at v0.32.0 (commit d051f46) across six categories
(dead code, error handling, test coverage, security, async correctness, config
hygiene), plus an empirical performance benchmark against seeded databases of
5,000 and 20,000 documents.

## Critical

### 1. `restore_backup` can destroy the live knowledge base
`emdx/services/backup_service.py:154-195` — restore writes directly onto the live
DB (`sqlite3.connect(self.db_path)` + `src.backup(dst)`) with no pre-restore
safety copy, no integrity check of the backup, and no atomic replace. A corrupt
or truncated backup file, or an interruption mid-restore, leaves the real DB
partially overwritten. The exception is caught and reported as "Restore failed"
— after the damage is done.

**Fix:** restore into a temp file in the same directory, run
`PRAGMA integrity_check` on it, snapshot the current DB to `*.pre-restore`, then
`os.replace()` the validated copy over `self.db_path`. On any failure, leave the
live DB untouched.

### 2. Fabricated Haiku model ID breaks `maintain entities --llm`
`emdx/services/entity_service.py:787` — `_MODEL_MAP["haiku"] = "claude-haiku-4-5-20250315"`.
That ID does not exist (the real one is `claude-haiku-4-5-20251001`). It is fed
directly into `claude --print --model ...`, so `emdx maintain entities --llm
--model haiku` 404s — and `haiku` is this module's default/fallback.

**Fix:** use the dateless alias `claude-haiku-4-5` (stable across snapshot bumps).

### 3. Deprecated Opus/Sonnet 4.0 IDs likely already dead
`emdx/services/entity_service.py:788-789` and `emdx/services/synthesis_service.py:257`
pin `claude-sonnet-4-20250514` / `claude-opus-4-20250514`, whose announced
retirement date (2026-06-15) has passed. `emdx labs compact` (OpusSynthesisService
default) and entity-service opus/sonnet paths may already be failing.

**Fix:** move to dateless aliases (`claude-sonnet-4-5`, `claude-opus-4-6` or
newer) and centralize (see Warning #5).

## Warnings

### 4. `get_subprocess_env` was never wired in — nested Claude sessions can break
`emdx/utils/environment.py:9` defines `get_subprocess_env()` (scrubs `CLAUDECODE`
for nested `claude` subprocess calls) but nothing imports it. Meanwhile
`ask_service.py:253` and `entity_service.py:993` run `subprocess.run(["claude",...])`
with no `env=`. The helper looks like it was written for exactly these call
sites and never connected.

**Fix:** pass `env=get_subprocess_env()` at both call sites (or delete the module
if the scrub is obsolete).

### 5. Two competing model-alias maps disagree — and the "real" one is orphaned
`emdx/config/cli_config.py` (`MODEL_ALIASES`: `opus` → `claude-opus-4-6`) is
imported by nothing; `emdx/services/entity_service.py:786` (`_MODEL_MAP`:
`opus` → `claude-opus-4-20250514`) is what actually runs. The same shorthand
resolves to different models depending on code path, and hardcoded
`DEFAULT_MODEL` literals are scattered across 8+ files
(`synthesis_service.py:121,257`, `ask_service.py:286`,
`wiki_synthesis_service.py:46`, `wiki_quality_service.py:470`, `core.py:1334`,
`briefing.py:477`, `wiki.py:615`).

**Fix:** make `cli_config.resolve_model_version` the single source of truth,
import it everywhere, delete `_MODEL_MAP` — or delete `cli_config.py` and pick
one place. Also stale: `compact.py:217` help text advertises the wrong default;
`entity_service.py:793` opus cost constants reflect legacy pricing (~3× high).

### 6. `--db-url` / `EMDX_DATABASE_URL` is dead config that tests rely on
`emdx/main.py:146` declares the option but nothing reads `db_url`;
`get_db_path()` only honors `EMDX_TEST_DB`/`EMDX_DB`. `tests/test_stale.py:38`
and `tests/test_commands_stale.py:137,189,238` set `EMDX_DATABASE_URL` expecting
isolation they do not get.

**Fix:** delete the option and switch those tests to `EMDX_TEST_DB` (or wire it
into the documented priority chain).

### 7. Discarded `asyncio.create_task` — tasks can be garbage-collected mid-flight
Three sites hold no reference to the created task, so asyncio's weak reference
allows silent GC/cancellation:
- `emdx/ui/browser_container.py:356` (palette result handling)
- `emdx/ui/command_palette/palette_screen.py:207` (result render)
- `emdx/ui/command_palette/palette_screen.py:254` (debounced search — also
  allows concurrent stale searches to clobber `_state.results`;
  `palette_presenter.py:140-178`)

**Fix:** `self.run_worker(coro, exclusive=True)` at each site — the exclusive
worker also fixes the stale-search race.

### 8. Blocking N+1 DB work on the TUI event loop
`emdx/ui/activity/activity_data.py:30-91` — `load_all` is `async` but does
synchronous SQLite work including a per-document `get_document_tags()` loop
(up to 100 queries), awaited on the event loop from `activity_view.py:392,767`.
The periodic refresh stalls the whole TUI. Same pattern (smaller) in
`knowledge_graph_panel.py:117-138` on every selection change.

**Fix:** `await asyncio.to_thread(...)` the sync loader (mirroring
`hybrid_search.search_unified`) and/or batch tags into one query.

### 9. Backup/cloud-backup error handling gaps
- `cloud_backup_service.py:72-116` — `upload`/`download` catch only
  `RuntimeError`; provider `OSError`/HTTP errors crash with raw tracebacks;
  `list_backups`/`delete` have no handling at all.
- `backup_providers/github.py:155-158` — `gh` not installed / not authenticated
  is swallowed into `return []`, so `cloud-backup --list` reports "No cloud
  backups found" to a user whose backups are fine.
- `backup_service.py:157-175` — decompressed temp DB leaks when restore raises.

### 10. Untested data-mutating hot spots
- Data-transforming migrations have no tests: `migration_042` (emoji-tag
  conversion), `migration_037` (FK table rebuild), `migration_019` backfill,
  `migration_032` (`emdx/database/migrations.py`). A rebuild bug would silently
  drop rows.
- `emdx/models/tasks.py:469` `_would_cycle` — transitive cycles (A→B→C→A) never
  exercised against a real DB; this guards dependency-graph integrity.
- `emdx/database/documents.py:614` `find_supersede_candidate` — zero tests; a
  false positive silently supersedes the wrong document.
- Command-layer gaps: `maintain_index.py` link/wikify/entities wrappers,
  `labs_ask.py` watch add/list/check/remove.

Coverage elsewhere is good: core save/find/view/edit/delete, task state
transitions, FTS escaping, trash/restore, and the wiki surface are well tested.

## Security: clean

No confirmed findings at any severity. Verified: all SQL parameterized (dynamic
fragments come from whitelists/hardcoded tokens only); `escape_fts5_query`
correctly neutralizes FTS5 operators; no `shell=True`/`os.system` anywhere —
all subprocess calls are list-form argv; wiki export slugs strip `.`/`/` so no
traversal; no secrets, no pickle/eval, temp files via `mkstemp` (0600).
Info-only: three f-string `LIMIT {limit}` sites (int-typed, not injectable —
bind for consistency: `wiki.py:1009`, `duplicate_detector.py:214`,
`auto_tagger.py:311`).

## Performance (measured, not read)

Method: seeded a real DB via the production code path (5,000 docs, mixed sizes,
54.6 MB → then grown to 20,000 docs), timed full CLI invocations (5 runs,
median) and in-process query latency.

| Command (5k docs) | median |
|---|---|
| `--help` (cold start floor) | 219ms |
| `find "sqlite migration"` (FTS) | 200ms |
| `find --all` | 171ms |
| `find --tags analysis` | 181ms |
| `view <id>` | 197ms |
| `save` (stdin, auto-link on) | 318ms |
| `task ready` | 269ms |
| `status` | 199ms |
| `prime --brief` | 166ms |

Key facts:
- **Everything is startup-bound.** The ~200ms floor is Python interpreter +
  imports; `emdx.main` costs ~92ms cumulative, of which
  `emdx.commands.briefing` alone is ~33ms and typer/rich/click ~32ms
  (`python -X importtime`). Query work is a small fraction on top.
- **The data layer scales fine.** In-process FTS: 16ms median at 5k docs, 29ms
  at 20k docs — sublinear-ish growth, nowhere near a problem. `save_document`
  DB layer: 12ms.
- **`save` overhead decomposes as:** ~200ms startup + ~50ms save path + ~60ms
  auto-link (323ms vs 262ms with `--no-auto-link` at 20k docs, no embedding
  index).
- **Bulk ingest** through `save_document` runs ~60 docs/s for large docs,
  ~90 docs/s for small ones (FTS indexing included).

Implication: per-command latency is fine for humans; for agents making many
calls, the highest-leverage wins are lazy-importing `briefing` (and any other
eagerly imported command modules) out of `emdx.main`, not query optimization.
`task ready` (+50ms over floor) is the only query-side outlier worth a look.
